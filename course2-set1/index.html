データアーカイブ

問1

不適切なアーカイビングの周期性は、パフォーマンスにどのような影響を与えますか？

  • A不必要にインデックス作成が速くなる
  • Bプライマリカタログの肥大化・遅延を引き起こす
  • Cアーカイブされたデータにアクセスできなくなる
  • Dデータのアーカイブを妨げる

正解：B

アーカイブの周期が不適切（頻度が低すぎるなど）だと、古い不要データがプライマリカタログに残り続けてテーブルが肥大化します。その結果、アクティブデータへのクエリが遅くなり、パフォーマンスが低下します。

問2

データアーカイブはどのようにITコストを削減しますか？

  • Aメインストレージの使用量を増加させる
  • Bすべてのアクティブなデータを圧縮する
  • Cバックアップとリカバリの時間を短縮する
  • D重要なシステムログを削除する

正解：C

アーカイブによってプライマリカタログのアクティブデータ量が減るため、バックアップやリカバリにかかる時間が短縮され、運用コストとダウンタイムを削減できます。ストレージ使用量の増加やログの削除は、コスト削減どころか逆効果や運用リスクにつながります。

問3

データアーカイブとは何ですか？

  • Aデータを二次記憶装置（セカンダリストレージ）に移動すること
  • Bすべてのデータをメインカタログに保存すること
  • C将来の使用のために現在のデータを暗号化すること
  • D関連のないデータを永久に削除すること

正解：A

データアーカイブとは、参照頻度の低くなったデータをメインカタログから二次記憶装置（セカンダリストレージ）へ移動することです。削除ではないため、必要になれば参照や復元が可能です。

問4

データアーカイブがパフォーマンスにとって重要な理由は何ですか？

  • A古いデータと新しいデータを統合する
  • Bすべてのデータベースインデックスを高速化する
  • Cプライマリ（主要）データのボリュームを最小限に抑える
  • D使われていないアプリケーションを削除する

正解：C

アーカイブによりプライマリデータのボリュームを最小限に抑えると、クエリが走査するアクティブデータが減り、実行時パフォーマンスが向上します。これがアーカイブがパフォーマンスにとって重要な理由です。

問5

データアーカイブのエンドユーザーにとっての主な利点は何ですか？

  • A暗号化されたアーカイブデータへのアクセス
  • B実行時（ランタイム）パフォーマンスの向上
  • Cデータ取得の追加オプション
  • Dデータ圧縮の削減

正解：B

エンドユーザーが直接享受するメリットは、アクティブデータが軽くなることによる画面表示や処理の高速化、つまり実行時（ランタイム）パフォーマンスの向上です。

問6

設計段階でデータ成長の予測を行うことは、どのように役立ちますか？

  • Aアーカイブの必要性をなくす
  • Bアプリケーションのパフォーマンス問題を回避する
  • Cデータベース監視の必要性を減らす
  • Dすべてのアクティブデータを永久に保持することを保証する

正解：B

設計段階でデータ成長を予測しておけば、アーカイブ戦略やインデックス設計を事前に計画でき、将来のアプリケーションのパフォーマンス問題を未然に回避できます。

問7

アーカイブカタログにインデックスが重要なのはなぜですか？

  • Aより高速な検索性能を可能にするため
  • B自動的な削除を可能にするため
  • Cレコードをユーザー入力なしで削除するため
  • Dシステムリソースの使用量を減らすため

正解：A

アーカイブカタログのデータも検索されることがあります。インデックスを付与しておくことで、大量のアーカイブデータに対しても高速な検索性能を維持できます。

問8

アーカイブされたデータのデータベースメンテナンス計画のメリットは何ですか？

  • Aインデックスの必要性を減らす
  • Bアーカイブされたデータを自動的に削除する
  • Cすべてのクエリの遅延を防止する
  • Dデータベースの健全性と最適化を保証する

正解：D

アーカイブデータに対しても定期的なメンテナンス計画（統計情報の更新やインデックスの再編成など）を実施することで、データベースの健全性と最適化が保証されます。

問9

アーカイブのタイマーはなぜピーク時間外に実行すべきですか？

  • Aデータベースの負荷を軽減するため
  • Bユーザーアクセスを簡素化するため
  • Cインデックスのパフォーマンスを向上させるため
  • Dシステムのダウンタイムを減らすため

正解：A

アーカイブ処理は大量のレコードの移動・削除を伴い、データベースに高い負荷をかけます。ユーザー利用の少ないピーク時間外に Timer を実行することで、データベースの負荷を軽減し本番アプリへの影響を最小化できます。

問10

アーカイブカタログのためにエンティティをコピーするとき、削除されるプロパティは何ですか？

  • Aオートナンバー（Auto Number）プロパティ
  • BIsActive フラグ
  • CIsArchived 属性
  • Dプライマリキー

正解：A

アーカイブ先の Entity では元レコードの識別子の値をそのまま保持する必要があるため、値を自動採番するオートナンバー（Auto Number）プロパティは削除（無効化）します。自動採番のままだと元の Id と食い違ってしまいます。

問11

アーカイブカタログでパージ（削除・整理）が行われない場合の影響は何ですか？

  • Aアーカイブされたデータは容量が増え続ける
  • Bデータは自動的に圧縮される
  • Cデータベースのパフォーマンスが向上する
  • Dアーカイブされたデータはアクセス可能なままである

正解：A

アーカイブはデータの移動であって削除ではないため、不要になった古いアーカイブを整理するパージを行わないと、アーカイブカタログの容量は増え続けます。

問12

言及されているアーカイブのバリアント2種類は何ですか？

  • A内部および外部アーカイブ
  • B軽量（ライト）および履歴（ヒストリカル）アーカイブ
  • C履歴および一時的アーカイブ
  • Dクイックおよびヘビーアーカイブ

正解：B

アーカイブには2つのバリアントがあります。検索・復元が可能な軽量（Light）アーカイブと、監査・長期保管向けにJSON形式などで保存する履歴（Historical）アーカイブです。

問13

ライトアーカイビングの主な特徴は何ですか？

  • Aアーカイブされたデータは即座に削除される
  • Bアーカイブされたデータは外部にのみ保存される
  • Cアーカイブされたデータは永続的にアクセスできない
  • Dアーカイブされたデータは検索・復元が可能

正解：D

ライトアーカイビングの主な特徴は、データを削除せず二次ストレージに保持するため、必要に応じて検索や復元が可能なことです。即時削除やアクセス不能になるわけではありません。

問14

アーカイブされたエンティティは、なぜ履歴アーカイブでJSON形式で保存されるのですか？

  • Aセキュリティリスクを減らすため
  • B一括削除を可能にするため
  • Cすべてのクエリを最適化するため
  • Dメタデータのタグ付けを可能にするため

正解：D

履歴アーカイブでJSON形式を使うのは、スキーマに依存しない柔軟なメタデータのタグ付けが可能になるためです。これにより、長期保管された履歴データを後から識別・検索しやすくなります。

問15

「IsArchived」コントロール列の目的は何ですか？

  • Aデータをアーカイブ後に削除する
  • Bレコードを削除のためにマークする
  • Cデータをアクティブとしてマークする
  • Dレコードをアーカイブ済みとしてフラグを立てる

正解：D

IsArchived はブール型のコントロール列で、そのレコードがアーカイブ済みであることを示すフラグです。これによりアクティブなレコードとアーカイブ対象のレコードを区別できます。

問16

タイムスタンプに基づくアーカイブのためにエンティティに追加される列はどれですか？

  • AIsDraft（下書き）
  • BLastUpdatedOn（最終更新日時）
  • CIsArchived（アーカイブ済み）
  • DIsActive（有効）

正解：B

タイムスタンプに基づいてアーカイブ対象を判定するには、レコードの最終更新日時を保持する LastUpdatedOn 列を Entity に追加します。一定期間更新されていないレコードをアーカイブ対象として抽出できます。

問17

「メインカタログ」は何を指しますか？

  • A主要（プライマリ）ストレージ
  • B外部API
  • C二次（セカンダリ）ストレージ
  • D履歴アーカイブ

正解：A

メインカタログとは、アプリケーションが日常的に読み書きするプライマリストレージ（本番データベース）を指します。アーカイブは、ここから二次ストレージへデータを移す操作です。

問18

OutSystems Cloud では、アーカイブされたデータはどこに保存できますか？

  • A外部データベースまたは OutSystems データベース
  • Bオンプレミスのサーバーのみ
  • CAPI のみ
  • DJSON ファイルのみ

正解：A

OutSystems Cloud では、アーカイブデータの保存先として外部データベース、または OutSystems のデータベースのいずれかを選択できます。オンプレミス限定やJSONファイル限定ではありません。

データベース / クエリ / エンティティ

問19

外部データベースのエンティティでは何ができませんか？

  • ASQLクエリでエンティティの出力を行う
  • BAggregateで作成または削除アクションを実行する
  • C異なるカタログスキーマの OutSystems エンティティとの結合（ジョイン）

正解：C

外部データベースの Entity は、異なるカタログ／スキーマにある OutSystems の Entity と直接ジョインできません。結合が必要な場合は、まずデータを取得してからアプリケーション側で突き合わせる必要があります。

問20

データベースからエンティティをインポートする際、Import Entities from Database Wizard を使うときの注意点は何ですか？

  • A最初に Service Center でデータベース接続を設定する必要がある
  • B開発者は外部キーをエンティティの識別子タイプに明示的にマッピングしなければならない
  • C複合プライマリキーを持つエンティティは、サポート外のデータ型と同様にプラットフォームのデータ型へマッピングが必要
  • Dきちんと作成された拡張機能では、これらのエンティティをクエリし、Service Studio で定義したエンティティとジョインできる

正解：A

Import Entities from Database Wizard を使う前提として、まず Service Center で対象データベースへの接続（Database Connection）を設定しておく必要があります。接続がなければウィザードはテーブルを参照できません。

問21

Product（外部DBに定義、Extension経由）と OrderProduct（Service Studio内に定義、Productへの参照属性を持つ）があり、Productには何千件ものレコードがあります。この2つを結合しようとするとき、正しいのはどれですか？

  • A常にエンティティを個別にクエリし、ループでレコードをマッチさせるのが良い
  • B可能なら外部DBのテーブル全体を結合に使うのが常に推奨される。外部DBはこの操作に最適化されている
  • C可能なら、結合前に必要最小限のデータを外部DBから取得し、一時エンティティなどを使って行う
  • Dプラットフォームは、Service Studio定義エンティティが外部DB定義の参照エンティティを参照することを許可していない

正解：C

何千件もあるテーブルを丸ごと結合するとパフォーマンスが大きく劣化します。ベストプラクティスは、結合前に外部DBから必要最小限のデータだけを取得し、一時的な Entity などに格納してから結合することです。

問22

エンティティを静的エンティティ（Static Entity）に変換するとき、正しいのはどれですか？

  • A静的エンティティは元の属性を保持し、さらに Order、Label、Is_Active 属性も追加する
  • BService Studio でエンティティ内のすべてのレコードを取得できる
  • C静的エンティティは、エンティティのインデックスを保持しない

正解：A

Entity を Static Entity に変換すると、元の属性を保持したまま Order・Label・Is_Active などの属性が自動追加されます。ただし既存のDBレコードは自動では取り込まれないため、レコードの取得を期待する B は誤りです。

問23

属性が20個あるレコードのうち、3つの値だけを更新したい場合、最良の方法は何ですか？

  • A部分更新APIを使い、変更したいフィールドのみを修正する
  • B更新されたフィールドを保存前に検証する
  • Cバッチ更新でリソース使用量を最小化する
  • DSQLクエリの UPDATE を使い、3つのフィールドだけを更新する
  • EORMツールで特定フィールドだけをターゲットに更新する
  • Fレコードの値を更新後、CreateOrUpdate を使う

正解：D

Aggregate で取得して Update アクションで保存するとレコード全体（20属性）を書き戻すことになり無駄が多くなります。3つの値だけを更新するなら、対象列のみを書き換える SQL の UPDATE 文が最も効率的です。

問24

SQLクエリ（Advanced Query）を作成するとき、パフォーマンス問題を避けるために考慮すべきことは何ですか？

  • A必要な属性だけを返すビューを作成してクエリする
  • B返される属性だけを含む特定の出力 Structure（Output Structure）を作成する
  • C実行時エラーを避けるため SELECT {Entity}.* を使い、パフォーマンスを上げる
  • Dプラットフォームが自動でSQLを最適化し、使用属性だけをフェッチするので最適化は不要

正解：B

SQLクエリ（Advanced Query）は Aggregate と違い属性の自動最適化が行われないため、必要な属性だけを含む専用の Output Structure を作成するのがベストプラクティスです。SELECT {Entity}.* で全列を取得すると不要なデータまで転送され、パフォーマンスが低下します。

問25

SQLクエリで Expand Inline パラメーターを使用する理由として、最も適切なものは何ですか？

  • A必要な句をインジェクションすることで動的条件を実装するため
  • B通常のパラメーター使用による SQL インジェクションを防ぐため
  • CEncodeSql 関数とともに使い、SQL インジェクションを避けるため
  • Dクエリのいずれかの句を省略可能（オプション）にするため

正解：A

Expand Inline を有効にすると、パラメーターの中身がSQL文の一部としてそのまま展開されるため、条件句などを動的に組み立てられます。ただし値がエスケープされないため、SQLインジェクションのリスクに注意して使う必要があります。

問26

SQLクエリで Expand Inline パラメーターを使う際、最も良くないとされる方法はどれですか？

  • Aすべての選択肢が悪い方法である
  • BReplace 関数を使って手動で文字列のエンコーディングを行う
  • CEncodeSql で SQL パラメータの全内容をエンコードする 例: `myparameter = EncodeSql("WHERE surname = " + @myVariable1 + " OR name = " + @myVariable2)`
  • Dすべての値をラップして IN 句を作る 例: `values = EncodeSql(name1 + "," + name2 + "," + name)` → `IN (@values)`

正解：A

Replace による手動エンコード、句全体を EncodeSql でくるむ方法、複数の値をまとめてエンコードして IN 句を作る方法は、いずれも正しくエスケープされずSQLインジェクションを許す危険な実装です。したがって「すべての選択肢が悪い方法」が正解で、EncodeSql は個々の値に対して適用しなければなりません。

パフォーマンス / タイマー / プロセス

問27

Web画面でのみ使用される大きなJavaScriptコードは、どこに置くべきですか？

  • AモジュールのJavaScriptプロパティに置く。そこだけブラウザキャッシュを利用できるから
  • BWeb画面内のエスケープされていない式に置く。一度だけ使うので開発者にとって明示的だから
  • CWeb画面内のエスケープされていない式として使われる関数の戻り値として置く。再利用性が高まるから
  • DWeb画面のJavaScriptプロパティに置く。ブラウザキャッシュを利用できるから

正解：D

その画面でしか使わない大きなJavaScriptは、Web画面のJavaScriptプロパティに配置します。ここに置いたコードはブラウザにキャッシュされるため、式の中に直書きするよりパフォーマンスと保守性の両面で優れます。モジュール側に置くと使わない画面にも読み込まれてしまいます。

問28

高頻度で再利用されるAggregateは、常にサーバーアクション内で隔離するのが良いプラクティスである。

  • A真 — 抽象化を促進し、実装の詳細を隠すから
  • B偽 — コードの難読化につながる可能性があるから
  • C真 — 変更箇所が1つになり保守性が向上するから
  • D偽 — 使用される属性に基づく自動クエリ最適化を妨げる可能性があるから

正解：D

Aggregate は呼び出し元で実際に使われる属性だけをフェッチするよう自動最適化されます。Server Action 内に隔離すると呼び出し元での使用状況が伝わらず全属性を取得してしまい、この最適化が効かなくなる可能性があるため、常に隔離するのは良いプラクティスとはいえません。

問29

現在は問題なく動作しているアプリに、明日から多数の新規ユーザーがアクセスし、データアクション（画像）の使用が増えます。最大レコード数50のAggregateとその出力リストがあります。遅延やタイムアウトを避ける最良の方法は？

  • AAggregateにキャッシュを使用する
  • Bタイマーを作成し、少量のバッチでレコードを処理する
  • C最大レコード数を変更する

正解：A

多数のユーザーが同じデータを繰り返し要求する状況では、Aggregate のキャッシュ（Cache in Minutes）を有効にするのが最良です。データベースへの問い合わせ回数が減り、遅延やタイムアウトを回避できます。

問30

ClientEnergyData エンティティに顧客のエネルギー消費データが保存されています。タイマーで全レコードを繰り返し処理し、処理後にSQLで全レコードを削除したい。タイムアウトを避けて効率的に処理する最良のアプローチは？

  • A実行時間を制御し、タイムアウト前に停止させる
  • Bバッチ処理を行い各バッチをコミットする。各バッチ後に経過時間を監視し、タイムアウト直前になったらタイマーを再起動する
  • C毎回固定数のレコードだけを処理する
  • DFor-Eachループで削除し明示的にコミットして、タイムアウト時に全変更がDBに反映されていることを保証する

正解：B

大量データの処理は、バッチ単位で処理してバッチごとにコミットし、各バッチ後に経過時間を監視するのが定石です。タイムアウトが近づいたら Timer 自身を Wake で再起動して続きを次回実行に引き継ぐことで、タイムアウトを回避しつつ全件を確実に処理できます。

問31

少量のデータ移行を行いたい場合、最適な方法はどれですか？

  • ABPT（ビジネスプロセステクノロジー）
  • BLight Process（軽量プロセス）
  • CTimer（タイマー）

正解：C

少量データの移行には、シンプルに1回実行できる Timer が最適です。BPT や Light Process は業務プロセスや大量イベントの並列処理向けの仕組みで、単発の少量移行には過剰です。

問32

タイマー1がタイムアウトしてタイマー2を起動させたとき、タイマー2には何が起こりますか？

  • Aタイマー2は部分的に初期化されるが、タイマー1のタイムアウトで完全に実行できなくなる
  • B何も起こらない。タイマー2は3回リトライし、エラー／タイムアウトが発生するとそれ以降は起きなくなる
  • Cタイマー2はタイマー1が正常にリトライするまで待機状態になる
  • Dタイマー2は、タイマー1がタイムアウトしても無期限に非アクティブのままである
  • Eタイマー2は、タイマー1がトリガーしなくても一定遅延後に起きる
  • Fタイマー2は、タイマー1のタイムアウトに関係なく独立して動作するため起きる

正解：B

Timer1 がタイムアウトすると実行中のトランザクションがロールバックされ、その中で行った Timer2 の Wake（次回実行のスケジュール）も一緒に取り消されます。Timer は既定で数回再試行され、それでも失敗し続けると、Timer2 は起動しません。

問33

どのような状況でプロセス（Process）を起動できますか？

  • Aしかるべき動作フロー内だけで明示的に起動される
  • Bどのエンティティのレコード作成・更新時にも自動的に起動される
  • Cプロセスイベントを公開しているエンティティのレコードが作成されたときに自動的に起動される
  • Dエンティティのレコード更新時に自動的に、またはアクション内で明示的に起動される

正解：C

プロセスは、Expose Process Events を有効にした Entity にレコードが作成されたときに自動的に起動できます。すべての Entity で起動するわけではなく、更新時ではなく作成時である点がポイントです（なお、アクションから明示的に起動することも可能です）。

アーキテクチャ / モジュール設計

問34

次のうち、コアビジネス層（Core Business layer）に含まれるべきものはどれですか？

  • Aビジネスロジック
  • Bテーマ定義
  • Cユースケースを調整するプロセスワークフロー定義
  • Dダッシュボードを持つホームページ

正解：A

Core Business 層には、再利用されるビジネスロジックやコアとなる Entity が含まれます。テーマ定義やホームページはエンドユーザー（UI）層、ユースケース全体を調整するプロセスワークフローはオーケストレーション層に属します。

問35

次のうち、OutSystems が推奨するアーキテクチャのベストプラクティスでは「ない」ものはどれですか？

  • A基礎（Foundation）モジュールは Core Services モジュールへ上方参照を持てる
  • B基礎モジュールは UIパターンを持てない
  • Cどの選択肢も推奨されるベストプラクティスではない
  • D基礎モジュールは読み取り専用のコアエンティティを持つべきである

正解：A

Architecture Canvas では参照は常に上位層から下位層への一方向であり、Foundation モジュールが上位の Core Services を参照する上方参照は禁止です。したがって A がベストプラクティスに反する記述です。B と D は正しいプラクティスです。

問36

あるアプリが小規模から大規模に成長し、展開が遅く、一部モジュールにUI・DB定義・ロジックが混在しています。UIをDB／ロジックから分離したい。特に "Finance" モジュールについて、データを失わない最適な方法は？

  • AFinance を開いてクローンし、クローンを Finance_CS にリネーム。元モジュールのUIフローを削除し、DBエンティティ／アクションを削除。クローンを公開し、そのエンティティ／アクションを元モジュールで参照。最後に元モジュールを公開
  • B新規に「Finance」「Finance_CS」を作成。元からUIフローを新Financeへ、DBエンティティ／アクションを新Finance_CSへ移動。依存関係を作り、元を削除して新モジュールを公開
  • CFinance をクローンし、元を Finance_CS にリネーム。クローンを Finance にリネームし全DBエンティティ／アクションを削除。元を公開しエンティティ／アクションをクローンで参照。最後にクローンを公開
  • D元を Finance_CS にリネームし、新規に「Finance」を作成。Refactor コンポーネントの指示に従って元から画面を新 Finance へ移行

正解：C

データを失わないための鍵は、実データ（物理テーブル）を持つ元モジュールを残すことです。C では元モジュールを Finance_CS にリネームしてデータを保持し、クローンをUI用の Finance にして自身のDB定義を削除し、元の Entity を参照します。A のように元モジュール側のDBを削除するとデータが失われます。

問37

クライアント変数（Client Variable）について正しいのはどれですか？

  • Aいいえ、大きなテキスト値を持たせるのは良いプラクティスではない
  • Bログアウト時やブラウザを閉じたときに値をリセットする
  • Cパフォーマンス最適化のため最小限のデータを保持する
  • D再利用のため値を常にローカルストレージに保存する
  • E追加の安全性のためテキスト値を暗号化する
  • Fクライアント変数には短期間のセッションを使用する

正解：A

Client Variable はデバイス側に保持されるため、大きなテキスト値を格納するのは良いプラクティスではありません。パフォーマンスやストレージへの悪影響を避けるため、必要最小限のデータのみを保持すべきです。

統合 / API / ライフサイクル

問38

API を使用しているとき、ログがなかったり Service Center に表示される情報が少ないのはなぜですか？

  • Aロギングレベルを「Full」に設定しなければならない（APIのリクエスト／レスポンスの全情報が含まれるため最も完全）
  • Bエンドポイントが間違っている
  • Cロギングレベルを「Troubleshooting」に設定しなければならない

正解：A

API のリクエスト／レスポンスの内容まで含めた完全なログを Service Center に記録するには、ロギングレベルを「Full」に設定する必要があります。既定のレベルではエラーの概要程度しか記録されず、トラブルシュートに必要な詳細が表示されません。

問39

API のレスポンスで City フィールドが値を返さないのはなぜですか？（City と Test の Structure 例を参照）

  • ACity フィールドが null または欠落しているため返されない
  • BStructure 内の City フィールドの JSON 名が、API レスポンスの City と同じではない
  • Cエンドポイントが間違っている
  • DAPI の設定やマッピングに問題があり、City フィールドが正しく埋められない

正解：B

レスポンス用 Structure の属性名（または JSON Name プロパティ）が、実際のAPIレスポンスのキー名と一致していないと、デシリアライズ時にマッピングされず値が空になります。JSON Name を実際のレスポンスに合わせて修正する必要があります。

問40

既定では OutSystems は Service Actions へのリクエストのエラー詳細をログに記録せず、トラブルシュートが困難です。Service Actions のログ詳細レベルを変更するには？

  • AService Center でアプリケーションの詳細ページを開き、「Operation」タブで Service Actions のログレベルを設定する
  • Bログ詳細レベルは変更できない
  • CService Studio で「Start Debugging」を選び、ログ詳細レベルのドロップダウンで「Troubleshooting」を選択する
  • DService Center で **モジュール**の詳細ページを開き、「Operation」タブで Service Actions のログレベルを設定する

正解：D

Service Actions のログレベルは、Service Center で対象モジュールの詳細ページを開き、「Operation」タブで設定します。アプリケーション単位ではなくモジュール単位の設定である点に注意してください。これでリクエストのエラー詳細まで記録できるようになります。

問41

Web リファレンスに関して、次のうち正しい記述はどれですか？

  • AWeb サービスが削除された場合、参照モジュールを公開するとき Service Studio はエラーを表示する
  • BWeb リファレンスのアクションと、それを呼び出す画面アクションは同じトランザクションを使用する
  • CWeb サービスが削除された場合、参照モジュールを公開するとき Service Studio は警告を表示する
  • DWeb リファレンスの Timeout（秒数）プロパティは、画面アクションで Web リファレンスを使うときに設定する必要がある

正解：C

参照先の Web サービスが削除されていても、参照モジュールの公開自体は可能で、Service Studio はエラーではなく警告（warning）を表示します。なお、Web リファレンスの呼び出しは呼び出し元の画面アクションとは別のトランザクションで実行されます。

イベント / ライフサイクル / セキュリティ

問42

On Destroy イベントはいつ発生しますか？

  • Aコンポーネントの再レンダリング中に発生する
  • Bブラウザがウェブページのアンロードやウィンドウを閉じるときにトリガーされる
  • C画面またはブロックを破棄する前に、DOM から削除されるときに発生する
  • D親コンポーネントのすべての子コンポーネントが破棄された後に発生する

正解：C

On Destroy イベントは、画面または Web Block が破棄されて DOM から削除される直前に発生します。イベントリスナーの解除やタイマーの停止といったクリーンアップ処理に使用します。

問43

URLパラメータやフォームの隠しデータの操作などにより、ユーザーがアプリの機密情報を改ざんできないようにする方法は？

  • AHTTP セキュリティ（SSL）を有効にし、パラメータがURLに表示されるのを暗号化する
  • Bシステム内の活動を監査し、情報にアクセスしようとするユーザーを見つけ出す
  • C画面遷移時に NAVIGATE の代わりに SUBMIT を使う。POST でパラメータがURLに表示されないから
  • D機密情報は、画面の入力パラメータの代わりにセッション変数を使って隠す

正解：D

URLパラメータやフォームの隠しデータに置いた値は、クライアント側で自由に改ざんできます。機密情報は画面の入力パラメータとして受け渡すのではなく、サーバー側で管理される Session Variable に保持することで改ざんを防げます。SSLは通信経路の暗号化であり、ユーザー自身による値の改ざんは防げません。

CRUD / トランザクション

問44

1つのアクションが CRUD アクションのフローと、例外ハンドラーのフロー（Abort=Yes）に与える影響は？（Audit の CRUD アクションもある）

※原資料（問題集.txt / 完成版.md）は正解を B（Audit も作成されない）としていますが、これは公式ドキュメントの挙動と矛盾します。Abort Transaction はロールバック後にハンドラを『新しいトランザクション』で実行するため、ハンドラ内の Audit は残ります（正解 A）。コース1セット1 Q1（同一シナリオ・公式検証済み）、コース2セット2 Q24（監査レコードだけ追加）とも一致します。

  • Aロールバックが行われ、Audit エンティティにレコードが作成される
  • Bロールバックが行われ、レコードは作成されない

正解：A

Exception Handler の Abort Transaction=Yes は、それまでのCRUD変更をロールバックしたうえでトランザクションを終了し、新しいトランザクションを開始します。Exception Handler 内で実行される Audit のCRUDはこの新しいトランザクションでコミットされるため、主フローはロールバックされても Audit レコードは作成されます。
