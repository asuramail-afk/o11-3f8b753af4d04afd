データアーカイブ

問1

不適切なアーカイビングの周期性は、パフォーマンスにどのような影響を与えますか？

  • A不必要にインデックス作成が速くなる
  • Bプライマリカタログの肥大化・遅延を引き起こす
  • Cアーカイブされたデータにアクセスできなくなる
  • Dデータのアーカイブを妨げる

正解：B

アーカイブの頻度が低すぎたり不適切だと、不要な古いデータがプライマリ（メイン）カタログに残り続け、テーブルが肥大化します。その結果、アクティブなデータへのクエリが遅くなり、パフォーマンスが低下します。

問2

データアーカイブはどのようにITコストを削減しますか？

  • Aメインストレージの使用量を増加させる
  • Bすべてのアクティブなデータを圧縮する
  • Cバックアップとリカバリの時間を短縮する
  • D重要なシステムログを削除する

正解：C

- A) メインストレージの使用量が増えると、逆にコスト増につながります。
- B) 圧縮はストレージ効率を上げますが、直接のコスト削減にはつながりにくいです。
- C) 正解。アクティブデータ量が減ることでバックアップ／リカバリの時間が短縮され、運用コストとダウンタイムを削減できます。
- D) 重要なシステムログを削除すると、障害調査や監査に支障が出ます。

問3

データアーカイブとは何ですか？

  • Aデータを二次記憶装置（セカンダリストレージ）に移動すること
  • Bすべてのデータをメインカタログに保存すること
  • C将来の使用のために現在のデータを暗号化すること
  • D関連のないデータを永久に削除すること

正解：A

アーカイブとは、参照頻度の低いデータをメインカタログから二次記憶装置へ移すことです。削除ではないため、必要時には参照・復元が可能です。

問4

データアーカイブがパフォーマンスにとって重要な理由は何ですか？

  • A古いデータと新しいデータを統合する
  • Bすべてのデータベースインデックスを高速化する
  • Cプライマリ（主要）データのボリュームを最小限に抑える
  • D使われていないアプリケーションを削除する

正解：C

アーカイブによってメインカタログ上のデータ量を最小化することで、アクティブなクエリ対象が減り、実行時パフォーマンスが向上します。

問5

データアーカイブのエンドユーザーにとっての主な利点は何ですか？

  • A暗号化されたアーカイブデータへのアクセス
  • B実行時（ランタイム）パフォーマンスの向上
  • Cデータ取得の追加オプション
  • Dデータ圧縮の削減

正解：B

エンドユーザーが直接得るメリットは、アクティブデータが軽くなることによる画面表示・処理の高速化、すなわち実行時パフォーマンスの向上です。

問6

設計段階でデータ成長の予測を行うことは、どのように役立ちますか？

  • Aアーカイブの必要性をなくす
  • Bアプリケーションのパフォーマンス問題を回避する
  • Cデータベース監視の必要性を減らす
  • Dすべてのアクティブデータを永久に保持することを保証する

正解：B

早期にデータ成長量を予測しておくことで、あらかじめアーカイブ戦略やインデックス設計を計画でき、将来のパフォーマンス問題を未然に防げます。

問7

アーカイブカタログにインデックスが重要なのはなぜですか？

  • Aより高速な検索性能を可能にするため
  • B自動的な削除を可能にするため
  • Cレコードをユーザー入力なしで削除するため
  • Dシステムリソースの使用量を減らすため

正解：A

アーカイブカタログでもデータ検索は発生します。インデックスを付与することで、大量のアーカイブデータに対しても高速な検索性能を維持できます。

問8

アーカイブされたデータのデータベースメンテナンス計画のメリットは何ですか？

  • Aインデックスの必要性を減らす
  • Bアーカイブされたデータを自動的に削除する
  • Cすべてのクエリの遅延を防止する
  • Dデータベースの健全性と最適化を保証する

正解：D

定期的なメンテナンス計画により、アーカイブDBの統計情報更新や再編成が行われ、データベースの健全性と最適化が保たれます。

問9

アーカイブのタイマーはなぜピーク時間外に実行すべきですか？

  • Aデータベースの負荷を軽減するため
  • Bユーザーアクセスを簡素化するため
  • Cインデックスのパフォーマンスを向上させるため
  • Dシステムのダウンタイムを減らすため

正解：A

アーカイブ処理は大量のレコード移動・削除を伴うため負荷が高いです。ユーザー利用の少ないピーク時間外に実行することで、本番アプリへの影響を最小化します。

問10

アーカイブカタログのためにエンティティをコピーするとき、削除されるプロパティは何ですか？

  • Aオートナンバー（Auto Number）プロパティ
  • BIsActive フラグ
  • CIsArchived 属性
  • Dプライマリキー

正解：A

アーカイブ先へエンティティをコピーする際、元の識別子の値をそのまま保持する必要があるため、値を自動採番するオートナンバー設定は解除（削除）されます。

問11

アーカイブカタログでパージ（削除・整理）が行われない場合の影響は何ですか？

  • Aアーカイブされたデータは容量が増え続ける
  • Bデータは自動的に圧縮される
  • Cデータベースのパフォーマンスが向上する
  • Dアーカイブされたデータはアクセス可能なままである

正解：A

アーカイブは移動であり削除ではないため、パージ（不要になった古いアーカイブの整理）を行わないと、アーカイブカタログの容量が際限なく増え続けます。

問12

言及されているアーカイブのバリアント2種類は何ですか？

  • A内部および外部アーカイブ
  • B軽量（ライト）および履歴（ヒストリカル）アーカイブ
  • C履歴および一時的アーカイブ
  • Dクイックおよびヘビーアーカイブ

正解：B

アーカイブには、検索・復元が可能な「軽量（Light）アーカイブ」と、監査・長期保管向けにJSON等で保存する「履歴（Historical）アーカイブ」の2種類があります。

問13

ライトアーカイビングの主な特徴は何ですか？

  • Aアーカイブされたデータは即座に削除される
  • Bアーカイブされたデータは外部にのみ保存される
  • Cアーカイブされたデータは永続的にアクセスできない
  • Dアーカイブされたデータは検索・復元が可能

正解：D

ライトアーカイブの特徴は、データを削除せず二次ストレージに残すため、必要に応じて検索・復元ができる点です。

問14

アーカイブされたエンティティは、なぜ履歴アーカイブでJSON形式で保存されるのですか？

  • Aセキュリティリスクを減らすため
  • B一括削除を可能にするため
  • Cすべてのクエリを最適化するため
  • Dメタデータのタグ付けを可能にするため

正解：D

JSON形式で保存することで、スキーマに依存しない柔軟なメタデータのタグ付けが可能になり、長期保管された履歴データを後から識別・検索しやすくなります。

問15

「IsArchived」コントロール列の目的は何ですか？

  • Aデータをアーカイブ後に削除する
  • Bレコードを削除のためにマークする
  • Cデータをアクティブとしてマークする
  • Dレコードをアーカイブ済みとしてフラグを立てる

正解：D

IsArchived はブール型のコントロール列で、そのレコードがアーカイブ済みかどうかを示すフラグです。これによりアクティブデータとアーカイブ対象を区別します。

問16

タイムスタンプに基づくアーカイブのためにエンティティに追加される列はどれですか？

  • AIsDraft（下書き）
  • BLastUpdatedOn（最終更新日時）
  • CIsArchived（アーカイブ済み）
  • DIsActive（有効）

正解：B

タイムスタンプ（時刻）ベースでアーカイブ対象を判定するには、レコードの最終更新日時を保持する LastUpdatedOn 列が必要です。一定期間更新されていないレコードをアーカイブ対象とします。

問17

「メインカタログ」は何を指しますか？

  • A主要（プライマリ）ストレージ
  • B外部API
  • C二次（セカンダリ）ストレージ
  • D履歴アーカイブ

正解：A

メインカタログとは、アクティブなアプリケーションが日常的に読み書きするプライマリストレージ（本番データベース）を指します。アーカイブは、ここから二次ストレージへデータを移す操作です。

問18

OutSystems Cloud では、アーカイブされたデータはどこに保存できますか？

  • A外部データベースまたは OutSystems データベース
  • Bオンプレミスのサーバーのみ
  • CAPI のみ
  • DJSON ファイルのみ

正解：A

OutSystems Cloud では、アーカイブ先として外部データベース、または OutSystems 管理下のデータベースのいずれかを選べます。

データベース / クエリ / エンティティ

問19

外部データベースのエンティティでは何ができませんか？

  • ASQLクエリでエンティティの出力を行う
  • Bアグリゲートで作成または削除アクションを実行する
  • C異なるカタログスキーマの OutSystems エンティティとの結合（ジョイン）

正解：C

外部データベースのエンティティは、異なるカタログ／スキーマにある OutSystems エンティティと直接ジョインすることはできません。結合するには、まず必要なデータを取得してから内部で突き合わせる必要があります。

問20

データベースからエンティティをインポートする際、Import Entities from Database Wizard を使うときの注意点は何ですか？

  • A最初に Service Center でデータベース接続を設定する必要がある
  • B開発者は外部キーをエンティティの識別子タイプに明示的にマッピングしなければならない
  • C複合プライマリキーを持つエンティティは、サポート外のデータ型と同様にプラットフォームのデータ型へマッピングが必要
  • Dきちんと作成された拡張機能では、これらのエンティティをクエリし、Service Studio で定義したエンティティとジョインできる

正解：A

ウィザードでインポートする前提として、まず Service Center 側で対象データベースへの接続（Database Connection）を設定しておく必要があります。

問21

Product（外部DBに定義、Extension経由）と OrderProduct（Service Studio内に定義、Productへの参照属性を持つ）があり、Productには何千件ものレコードがあります。この2つを結合しようとするとき、正しいのはどれですか？

  • A常にエンティティを個別にクエリし、ループでレコードをマッチさせるのが良い
  • B可能なら外部DBのテーブル全体を結合に使うのが常に推奨される。外部DBはこの操作に最適化されている
  • C可能なら、結合前に必要最小限のデータを外部DBから取得し、一時エンティティなどを使って行う
  • Dプラットフォームは、Service Studio定義エンティティが外部DB定義の参照エンティティを参照することを許可していない

正解：C

大量レコードを丸ごと結合するとパフォーマンスが著しく劣化します。ベストプラクティスは、外部DBから必要最小限のデータだけを取得し、一時（ローカル）エンティティ等に落としてから結合することです。

問22

エンティティを静的エンティティ（Static Entity）に変換するとき、正しいのはどれですか？

  • A静的エンティティは元の属性を保持し、さらに Order、Label、Is_Active 属性も追加する
  • BService Studio でエンティティ内のすべてのレコードを取得できる
  • C静的エンティティは、エンティティのインデックスを保持しない

正解：B

静的エンティティはレコードがメタデータとして扱われ、Service Studio 上で全レコードを直接参照・取得できます。（注: Order／Label／Is_Active はレコードのプロパティであり、Aの説明は不正確です。）

問23

属性が20個あるレコードのうち、3つの値だけを更新したい場合、最良の方法は何ですか？

  • A部分更新APIを使い、変更したいフィールドのみを修正する
  • B更新されたフィールドを保存前に検証する
  • Cバッチ更新でリソース使用量を最小化する
  • DSQLクエリの UPDATE を使い、3つのフィールドだけを更新する
  • EORMツールで特定フィールドだけをターゲットに更新する
  • Fレコードの値を更新後、CreateOrUpdate を使う

正解：D

アグリゲート＋Update アクションはレコード全体（20属性）を書き戻すため無駄が多くなります。3つだけを更新するなら、対象列だけを書き換える SQL の UPDATE 文が最も効率的です。

問24

SQLクエリ（Advanced Query）を作成するとき、パフォーマンス問題を避けるために考慮すべきことは何ですか？

  • A必要な属性だけを返すビューを作成してクエリする
  • B返される属性だけを含む特定の出力構造（Output Structure）を作成する
  • C実行時エラーを避けるため SELECT {Entity}.* を使い、パフォーマンスを上げる
  • Dプラットフォームが自動でSQLを最適化し、使用属性だけをフェッチするので最適化は不要

正解：B

SQLクエリでは、必要な列だけを返す専用の出力構造を定義するのがベストです。`.*` で全列を取得すると不要なデータまで転送され、パフォーマンスが低下します。また、Advanced Query はアグリゲートと異なり自動で属性最適化されません。

問25

SQLクエリで Expand Inline パラメーターを使用する理由として、最も適切なものは何ですか？

  • A必要な句をインジェクションすることで動的条件を実装するため
  • B通常のパラメーター使用による SQL インジェクションを防ぐため
  • CEncodeSql 関数とともに使い、SQL インジェクションを避けるため
  • Dクエリのいずれかの句を省略可能（オプション）にするため

正解：A

Expand Inline は、パラメーターの中身を SQL 文の一部として展開（インジェクション）します。これにより、条件句そのものを動的に組み立てられます。ただしエスケープされないため、SQLインジェクションのリスクがある点に注意が必要です。

問26

SQLクエリで Expand Inline パラメーターを使う際、最も良くないとされる方法はどれですか？

  • Aすべての選択肢が悪い方法である
  • BReplace 関数を使って手動で文字列のエンコーディングを行う
  • CEncodeSql で SQL パラメータの全内容をエンコードする 例: `myparameter = EncodeSql("WHERE surname = " + @myVariable1 + " OR name = " + @myVariable2)`
  • Dすべての値をラップして IN 句を作る 例: `values = EncodeSql(name1 + "," + name2 + "," + name)` → `IN (@values)`

正解：A

手動エンコード、句全体を EncodeSql でくるむ方法、値をまとめて IN 句にする方法は、いずれも正しくエスケープされずインジェクションを許す危険な実装です。したがって「すべての選択肢が悪い方法」が正解です。EncodeSql は個々の値に対して正しく適用する必要があります。

パフォーマンス / タイマー / プロセス

問27

Web画面でのみ使用される大きなJavaScriptコードは、どこに置くべきですか？

  • AモジュールのJavaScriptプロパティに置く。そこだけブラウザキャッシュを利用できるから
  • BWeb画面内のエスケープされていない式に置く。一度だけ使うので開発者にとって明示的だから
  • CWeb画面内のエスケープされていない式として使われる関数の戻り値として置く。再利用性が高まるから
  • DWeb画面のJavaScriptプロパティに置く。ブラウザキャッシュを利用できるから

正解：D

その画面でしか使わないJSは、画面（Screen）のJavaScriptプロパティに配置します。これによりコードがまとめてブラウザキャッシュされ、式の中に直書きするより保守性・パフォーマンスとも優れます。

問28

高頻度で再利用されるアグリゲートは、常にサーバーアクション内で隔離するのが良いプラクティスである。

  • A真 — 抽象化を促進し、実装の詳細を隠すから
  • B偽 — コードの難読化につながる可能性があるから
  • C真 — 変更箇所が1つになり保守性が向上するから
  • D偽 — 使用される属性に基づく自動クエリ最適化を妨げる可能性があるから

正解：D

アグリゲートは「呼び出し元で実際に使われる属性」だけをフェッチするよう自動最適化されます。サーバーアクション内に隔離すると、呼び出し元の使用状況が伝わらず全属性を取得してしまい、この最適化が効かなくなる場合があります。

問29

現在は問題なく動作しているアプリに、明日から多数の新規ユーザーがアクセスし、データアクション（画像）の使用が増えます。最大レコード数50のアグリゲートとその出力リストがあります。遅延やタイムアウトを避ける最良の方法は？

  • Aアグリゲートにキャッシュを使用する
  • Bタイマーを作成し、少量のバッチでレコードを処理する
  • C最大レコード数を変更する

正解：A

同じデータが繰り返し要求される画面表示では、アグリゲートのキャッシュ（Cache in Minutes）を有効にすることで、DBへの問い合わせ回数を減らし遅延・タイムアウトを回避できます。

問30

ClientEnergyData エンティティに顧客のエネルギー消費データが保存されています。タイマーで全レコードを繰り返し処理し、処理後にSQLで全レコードを削除したい。タイムアウトを避けて効率的に処理する最良のアプローチは？

  • A実行時間を制御し、タイムアウト前に停止させる
  • Bバッチ処理を行い各バッチをコミットする。各バッチ後に経過時間を監視し、タイムアウト直前になったらタイマーを再起動する
  • C毎回固定数のレコードだけを処理する
  • DFor-Eachループで削除し明示的にコミットして、タイムアウト時に全変更がDBに反映されていることを保証する

正解：B

大量データはバッチ単位で処理・コミットし、各バッチ後に経過時間を監視するのが定石です。タイムアウト直前になったらタイマー自身を「Wake」で再起動し、続きを次回実行に引き継ぐことで、タイムアウトを回避しつつ全件を確実に処理できます。

問31

少量のデータ移行を行いたい場合、最適な方法はどれですか？

  • ABPT（ビジネスプロセステクノロジー）
  • BLight Process（軽量プロセス）
  • CTimer（タイマー）

正解：B

少量データの移行には、オーバーヘッドの小さい Light Process が適しています。BPT は複雑なワークフロー向けで重く、タイマーは定期・大量バッチ向けです。

問32

タイマー1がタイムアウトしてタイマー2を起動させたとき、タイマー2には何が起こりますか？

  • Aタイマー2は部分的に初期化されるが、タイマー1のタイムアウトで完全に実行できなくなる
  • B何も起こらない。タイマー2は3回リトライし、エラー／タイムアウトが発生するとそれ以降は起きなくなる
  • Cタイマー2はタイマー1が正常にリトライするまで待機状態になる
  • Dタイマー2は、タイマー1がタイムアウトしても無期限に非アクティブのままである
  • Eタイマー2は、タイマー1がトリガーしなくても一定遅延後に起きる
  • Fタイマー2は、タイマー1のタイムアウトに関係なく独立して動作するため起きる

正解：B

タイマー1がタイムアウトすると、そこから起動されるはずのタイマー2はトリガーされません。タイマー自体はエラー／タイムアウト時に最大3回リトライされますが、それでも失敗すると以降は起動されなくなります。

問33

どのような状況でプロセス（Process）を起動できますか？

  • Aしかるべき動作フロー内だけで明示的に起動される
  • Bどのエンティティのレコード作成・更新時にも自動的に起動される
  • Cプロセスイベントを公開しているエンティティのレコードが作成されたときに自動的に起動される
  • Dエンティティのレコード更新時に自動的に、またはアクション内で明示的に起動される

正解：C

プロセスは、「プロセスイベント（Process Events）」を公開しているエンティティに対してレコード作成が発生したときに自動起動されます（加えて、アクションから Launch Process で明示的に起動することも可能です）。

アーキテクチャ / モジュール設計

問34

次のうち、コアビジネス層（Core Business layer）に含まれるべきものはどれですか？

  • Aビジネスロジック
  • Bテーマ定義
  • Cユースケースを調整するプロセスワークフロー定義
  • Dダッシュボードを持つホームページ

正解：A

コアビジネス層には、再利用されるビジネスロジックやコアエンティティが含まれます。テーマ定義やホームページはエンドユーザー層（UI）、業務全体を調整するオーケストレーションはより上位の層に属します。

問35

次のうち、OutSystems が推奨するアーキテクチャのベストプラクティスでは「ない」ものはどれですか？

  • A基礎（Foundation）モジュールは Core Services モジュールへ上方参照を持てる
  • B基礎モジュールは UIパターンを持てない
  • Cどの選択肢も推奨されるベストプラクティスではない
  • D基礎モジュールは読み取り専用のコアエンティティを持つべきである

正解：C

基礎モジュールは上位層（Core Services 等）を参照してはならず（下方参照のみ）、UIパターンを持つことはできますし、必ずしも「読み取り専用コアエンティティ」を持つわけでもありません。したがって A・B・D すべてが誤りで、「どれも推奨ベストプラクティスではない」が正解です。

問36

あるアプリが小規模から大規模に成長し、展開が遅く、一部モジュールにUI・DB定義・ロジックが混在しています。UIをDB／ロジックから分離したい。特に "Finance" モジュールについて、データを失わない最適な方法は？

  • AFinance を開いてクローンし、クローンを Finance_CS にリネーム。元モジュールのUIフローを削除し、DBエンティティ／アクションを削除。クローンを公開し、そのエンティティ／アクションを元モジュールで参照。最後に元モジュールを公開
  • B新規に「Finance」「Finance_CS」を作成。元からUIフローを新Financeへ、DBエンティティ／アクションを新Finance_CSへ移動。依存関係を作り、元を削除して新モジュールを公開
  • CFinance をクローンし、元を Finance_CS にリネーム。クローンを Finance にリネームし全DBエンティティ／アクションを削除。元を公開しエンティティ／アクションをクローンで参照。最後にクローンを公開

問37

クライアント変数（Client Variable）について正しいのはどれですか？

  • Aいいえ、大きなテキスト値を持たせるのは良いプラクティスではない
  • Bログアウト時やブラウザを閉じたときに値をリセットする
  • Cパフォーマンス最適化のため最小限のデータを保持する
  • D再利用のため値を常にローカルストレージに保存する
  • E追加の安全性のためテキスト値を暗号化する
  • Fクライアント変数には短期間のセッションを使用する

正解：A

クライアント変数はデバイス側に保持され、リクエストのたびにやり取りされる場合があります。大きなテキスト値を格納するとパフォーマンスやストレージに悪影響が出るため、良いプラクティスではありません。最小限のデータのみ保持すべきです。

統合 / API / ライフサイクル

問38

API を使用しているとき、ログがなかったり Service Center に表示される情報が少ないのはなぜですか？

  • Aロギングレベルを「Full」に設定しなければならない（APIのリクエスト／レスポンスの全情報が含まれるため最も完全）
  • Bエンドポイントが間違っている
  • Cロギングレベルを「Troubleshooting」に設定しなければならない

正解：A

REST/SOAP API 呼び出しのリクエスト・レスポンス本文まで含めて記録するには、ログレベルを「Full」に設定する必要があります。既定ではエラー概要程度しか記録されず、トラブルシュートに必要な詳細が出ません。

問39

API のレスポンスで City フィールドが値を返さないのはなぜですか？（City と Test の構造例を参照）

  • ACity フィールドが null または欠落しているため返されない
  • B構造内の City フィールドの JSON 名が、API レスポンスの City と同じではない
  • Cエンドポイントが間違っている
  • DAPI の設定やマッピングに問題があり、City フィールドが正しく埋められない

正解：B

レスポンス構造の属性名（またはその JSON Name プロパティ）が、実際のAPIレスポンスのキー名と一致していないと、デシリアライズ時にマッピングされず値が空になります。JSON Name を実レスポンスに合わせる必要があります。

問40

既定では OutSystems は Service Actions へのリクエストのエラー詳細をログに記録せず、トラブルシュートが困難です。Service Actions のログ詳細レベルを変更するには？

  • AService Center でアプリケーションの詳細ページを開き、「Operation」タブで Service Actions のログレベルを設定する
  • Bログ詳細レベルは変更できない
  • CService Studio で「Start Debugging」を選び、ログ詳細レベルのドロップダウンで「Troubleshooting」を選択する
  • DService Center で **モジュール**の詳細ページを開き、「Operation」タブで Service Actions のログレベルを設定する

正解：D

ログレベルは Service Center 上で対象**モジュール**の詳細ページ →「Operation」タブから設定します（アプリケーション単位ではなくモジュール単位）。これでリクエストのエラー詳細まで記録できます。

問41

Web リファレンスに関して、次のうち正しい記述はどれですか？

  • AWeb サービスが削除された場合、参照モジュールを公開するとき Service Studio はエラーを表示する
  • BWeb リファレンスのアクションと、それを呼び出す画面アクションは同じトランザクションを使用する
  • CWeb サービスが削除された場合、参照モジュールを公開するとき Service Studio は警告を表示する

イベント / ライフサイクル / セキュリティ

問42

On Destroy イベントはいつ発生しますか？

  • Aコンポーネントの再レンダリング中に発生する
  • Bブラウザがウェブページのアンロードやウィンドウを閉じるときにトリガーされる
  • C画面またはブロックを破棄する前に、DOM から削除されるときに発生する
  • D親コンポーネントのすべての子コンポーネントが破棄された後に発生する

正解：C

On Destroy は、画面（Screen）またはブロック（Block）が破棄され DOM から取り除かれる直前に一度だけ実行されます。イベントリスナーの解除やタイマー停止など、後片付け処理に使います。

問43

URLパラメータやフォームの隠しデータの操作などにより、ユーザーがアプリの機密情報を改ざんできないようにする方法は？

  • AHTTP セキュリティ（SSL）を有効にし、パラメータがURLに表示されるのを暗号化する
  • Bシステム内の活動を監査し、情報にアクセスしようとするユーザーを見つけ出す
  • C画面遷移時に NAVIGATE の代わりに SUBMIT を使う。POST でパラメータがURLに表示されないから
  • D機密情報は、画面の入力パラメータの代わりにセッション変数を使って隠す

正解：D

URLやフォームの隠し項目に置いた値は、クライアント側で改ざん可能です。機密情報は画面の入力パラメータとして持ち回るのではなく、サーバー側のセッション変数に保持することで、クライアントからの改ざんを防げます。

CRUD / トランザクション

問44

1つのアクションが CRUD アクションのフローと、例外ハンドラーのフロー（Abort=Yes）に与える影響は？（Audit の CRUD アクションもある）

※原資料（問題集.txt / 完成版.md）は正解を B（Audit も作成されない）としていますが、これは公式ドキュメントの挙動と矛盾します。Abort Transaction はロールバック後にハンドラを『新しいトランザクション』で実行するため、ハンドラ内の Audit は残ります（正解 A）。コース1セット1 Q1（同一シナリオ・公式検証済み）、コース2セット2 Q24（監査レコードだけ追加）とも一致します。

  • Aロールバックが行われ、Audit エンティティにレコードが作成される
  • Bロールバックが行われ、レコードは作成されない

正解：A

例外ハンドラの Abort Transaction=Yes は、そこまでのCRUD変更（主フロー）をロールバックし、同時に現在のトランザクションを終了して新しいトランザクションを開始します。例外ハンドラ内で実行される Audit の CRUD はこの新トランザクションでコミットされるため、ロールバックは行われても Audit レコードは作成されます。
