Q1

複雑なプロセスモデルの管理が難しくなっています。最善の長期的な解決策は何ですか？

  • Aすべてのロジックをインラインで追加する
  • B条件をスキップして簡略化する
  • C出力をハードコードする
  • Dさらに多くのタイマーを追加する
  • E複数の小さな再利用可能なサーバーアクションとプロセスに分割する
  • F複雑なときにエラーを表示する

正解：E

ロジックを小さな再利用可能なServer Actionやプロセスに分割（モジュール化）すると、再利用性・テスト容易性・保守性が高まり、長期的に管理しやすくなります。インライン化やハードコードは一時しのぎにすぎず、複雑さを増やすだけです。

Q2

オプションの入力を空のままにするとアプリがクラッシュします。最もメンテナンスしやすい解決策は？

  • Aクラッシュ後にタイマーで確認する
  • B空のケースは無視する
  • C必須に設定する（Set required = true）
  • D常にアラートを表示する
  • Eロジックと UI に null または空のチェックを追加する
  • FCDT でデフォルト値を設定する

正解：E

ロジックとUIの両方にnull・空値チェックを追加すれば、オプション入力が欠けていても安全に処理でき、クラッシュを防げます。任意の入力を必須に変えるのは要件を変えてしまい、UXも損ないます。

Q3

OAuth 2.0 を使うサードパーティと連携しています。トークンはどこに安全に保存すべきですか？

  • AJavaScript 変数
  • Bセッション変数またはアクセス制限されたサイトプロパティ
  • CCDT 属性
  • DUI にトークンを表示する
  • E静的エンティティ
  • Fセッション出力

正解：B

トークンはユーザースコープのSession Variableや、アクセスを制限したSite Propertyなどサーバー側に保存し、クライアントに露出させないことが重要です。JavaScript変数やUIへの表示はトークン漏えいにつながるため危険です。

Q4

画面のAggregateがあまりに多くの行を取得し、ロード時間に影響しています。どう改善しますか？

  • Aすべての行をクエリしてから画面上でフィルターをかける
  • B静的リストを使用する
  • Cローディングスピナーを長く表示する
  • Dページネーションとサーバー側のフィルターを実装する
  • EWeb ブロックを使用してビューを制限する
  • Fロード後にタイマーを実行する

正解：D

ページネーション（Max Records）とサーバー側フィルターをAggregateに実装し、データベースクエリの段階で取得件数を絞ることで、負荷が下がりロードが速くなります。全件取得してから画面側でフィルターするのは最も非効率な方法です。

Q5

外部データを1時間ごとに同期し、管理者が上書きできるようにしたい。スケーラブルな解決策は？

  • Aボタンのみ追加
  • B負荷が高い方法
  • Cスケーラブルでない方法
  • Dタイマーと管理者ロールに基づくオーバーライドトリガーを使用する
  • E各画面にロジックを追加
  • F警告のみ表示する

正解：D

定期同期はTimerで自動化し、管理者による上書きはロールチェック（Check<Role>）付きのトリガーで提供すれば、自動処理と手動オーバーライドを両立できるスケーラブルな構成になります。

Q6

割引の計算ロジックを複数のモジュールで一元管理する、最もきれいな方法は？

  • A画面ごとに式を追加
  • B共有モジュールに公開（Public）サーバーアクションを作成する
  • C動的でない方法
  • D静的エンティティを追加
  • E関連のないロジック
  • Fレイアウトに追加

正解：B

割引計算を共有ModuleのPublicなServer Actionに集約すると、ロジックを一元管理でき、各モジュールから再利用できます。画面ごとに式を複製すると重複やバージョンずれの原因になります。

Q7

一時的なフィルター設定をセッション中に保存したい。最も適切なストレージは？

  • ACDTs（Client Data Types）
  • Bエンティティ（Entities）
  • C静的エンティティ（Static Entities）
  • Dローカル変数（Local Variables）
  • Eセッション変数（Session Variables）
  • FWeb ブロック（Web Blocks）

正解：E

Session Variableはユーザーのセッション中ずっと値を保持できるため、一時的なフィルター設定の保存に最適です。Local Variableは画面やアクション内でしか保持されず、Entityへの保存は一時データには過剰です。

Q8

ユーザーが自分のプロフィールは編集でき、他ユーザーのレコードにはアクセスできないようにする最適な実装は？

  • Aログインユーザーの ID でAggregateをフィルタリングし、保存ロジックも制限する
  • B保存後にエラーを表示
  • Cロールエンティティを使用してフィルタリング
  • Dエンティティを追加して検証
  • Eすべて表示し、JavaScript で保存をブロック
  • Fタイマーを使ってアクセスを確認

正解：A

読み取り（Aggregate）と書き込み（保存ロジック）の両方をログインユーザーのID（GetUserId()）で制限することで、ユーザーが自分のデータだけを閲覧・編集できるよう安全に強制できます。JavaScriptでの制御だけではバイパスされる恐れがあります。

Q9

ユーザーが機微な画面へブックマーク（直接 URL アクセス）するのを防ぐには？

  • AJavaScript で表示/非表示を切り替える
  • Bロールチェックを実装し、権限のないアクセスをリダイレクトする
  • Cロード後にエラーを表示
  • D静的エンティティを使用
  • Eスタイルに hidden を追加
  • Fセッションのみをチェック

正解：B

サーバー側でロールチェックを行い、権限のないアクセスをログイン画面や権限なし画面へリダイレクトするのが正しい方法です。UIの非表示やJavaScriptによる制御はURL直接アクセスで簡単にバイパスされます。

Q10

API がペイロードのフォーマット問題でリクエストを拒否しています。最初に確認すべきことは？

  • Aエンティティをそのまま完全に送信する
  • Bコール前に成功を表示する
  • C静的エンティティのフォーマット
  • D入力/出力の構造が API 仕様と一致していることを確認する
  • Eリクエスト間の遅延ロジック
  • Fタイマーを追加してリトライする

正解：D

API連携でペイロードが拒否される場合、まず入力・出力のStructureがAPI仕様（スキーマ）と一致しているかを確認します。Timerでのリトライや遅延では、フォーマット不一致という根本原因は解決しません。

Q11

変更がほとんどない参照データを事前に読み込みたい。効率的に行うには？

  • A開発時のみ表示
  • Bすべての画面で毎回ロード
  • CStatic Entity を使用
  • DJS 変数を使用
  • Eセッション変数またはアプリケーションスコープの変数でキャッシュする
  • F条件ごとに Web ブロックを追加

正解：E

ほとんど変わらない参照データは、Session Variableやアプリケーションスコープの変数（またはAggregateのCache in Minutes）でキャッシュすることで、繰り返しのロードを避けてパフォーマンスを向上できます。

Q12

API を呼び出してデータを変換し、保存する必要があります。最適なアーキテクチャのアプローチは？

  • Aサーバーアクションを使って API を呼び出し、データを変換し、エンティティに保存する
  • BAPI の結果を画面にだけ表示する
  • CUI 上で API を実行する
  • D画面のロード時にタイマーを使う
  • E静的エンティティに追加する
  • FJavaScript を使って呼び出し、REST を呼ぶ

正解：A

API呼び出し・データ変換・Entityへの保存という一連の処理をServer Actionにカプセル化するのが最適です。サーバー側で完結させることで安全かつクリーンに統合でき、再利用と保守も容易になります。

Q13

REST エンドポイントを公開しつつ、内部ネットワークからのアクセスに限定したい。最も効果的な方法は？

  • Aタイマーを実行してから呼び出す
  • Bトークンだけをチェックする
  • Cロールチェック（Role check）を使う
  • D内部専用のテキストを表示する
  • E環境設定と IP 制限をネットワークまたはファイアウォール設定で行う
  • Fアプリのロジックだけで制御する

正解：E

ネットワークスコープの制限は、環境設定やファイアウォールでのIP制限（および公開APIのInternal Access Only設定）で行うのが最も効果的です。アプリのロジックやロールチェックではネットワーク境界そのものは制御できません。

Q14

REST API が予期しない構造を返したときに画面がクラッシュします。改善するには？

※消費（Consume）REST のコールバックの正式名は 'OnAfterResponse'（原資料の『OnResponse』はこれを指します）。非 2xx や構造不一致は既定で例外になるため、例外ハンドラや OnAfterResponse で処理・フォールバックします。

  • AUI にデフォルト値を追加する
  • Bレスポンスのコールバックでエラー処理とフォールバックロジックを実装する
  • C生の JSON を表示する
  • DJS のフォールバックを実行する
  • Eタイマーで毎回リトライする
  • F『もう一度試す』だけを表示する

正解：B

RESTレスポンスのコールバックでエラー処理とフォールバックロジックを実装すれば、想定外の構造が返ってきてもクラッシュせずに安全に処理を継続できます。生のJSON表示やTimerでのリトライは根本的な対策になりません。

Q15

マルチテナントアプリで、テナントごとに異なるテーマを有効にするには？

  • Aすべてのテーマを表示し、トグルで切り替える
  • B十分に動的でない方法
  • Cテナントのコンテキストに基づき、テーマ/スタイルクラスを条件付きでロードする
  • Dメンテナンス性が低い方法
  • EJavaScript のトグルやタイマーを追加して切り替える

正解：C

現在のテナントのコンテキストに基づいてテーマやスタイルクラスを条件付きでロードすることで、テナントごとの見た目を動的かつスケーラブルに切り替えられます。

Q16

ユーザーロールに基づいてモジュールの機能へのアクセスを制限します。どこでチェックすべき？

  • ACSS の表示/非表示を使う
  • Bログインメールに頼る
  • CJS を使って要素を隠す
  • Dロール不一致の場合にアラートを表示する
  • Eロジック層で Check<Role>() を使ってアクション実行前に確認する
  • FUI（ユーザーインターフェース）だけで制御する

正解：E

アクセス制御はサーバー側のロジック層で、アクション実行前にCheck<Role>()で検証するのが正解です。UIやCSS、JavaScriptでの非表示は簡単にバイパスされるため、セキュリティ対策にはなりません。

Q17

動的データを使用してチャートを作成します。どのコンポーネントを使うべき？

  • AOutSystems のチャートを使用し、Aggregateまたはレコードのリストにバインドする
  • Bテーブルを追加して色付けする
  • CWeb ブロックを使用する
  • D静的な画像を表示する
  • EJavaScript のみでチャートを実行する
  • Fタイマーを使ったビジュアライゼーション

正解：A

OutSystems内蔵のChartsコンポーネントを使い、Aggregateやレコードのリストにバインドすれば、データに応じた動的なチャートを簡単に表示できます。静的画像や独自JavaScriptによる実装は保守性に劣ります。

Q18

レコードの作成と編集は可能にし、削除はできないようにする画面を作る最良の方法は？

  • Aタイマーを設定して削除を行う
  • B削除ボタンを表示するが無効化しておく
  • C静的エンティティを使用する
  • D削除後にエラーを表示する
  • Eエンティティアクションを作成・更新だけに使い、削除のロジックを（UI とロジックの両方で）持たせない
  • F削除を許可し、ログを記録する

正解：E

作成・更新用のEntityアクションだけを使い、削除のロジックをUIとサーバーロジックの両方から排除することで、「削除不可」というビジネス制約を確実に強制できます。削除ボタンを無効化するだけでは、ロジック側に削除処理が残る限り安全とはいえません。

Q19

複雑なフォームで、複数の依存ドロップダウンがあります。2番目のドロップダウンに適切な値だけを表示させるには？

  • A両方のリストを完全にロードする
  • B静的エンティティのみを使用する
  • CJavaScript でフィルタリングする
  • D選択後にタイマーを実行する
  • E1つ目のドロップダウンの選択値に基づいてフィルタリングした動的なAggregateを使用する
  • F完全なリストを表示し、関係のないものは非表示にする

正解：E

2番目のドロップダウンのAggregateを1番目の選択値でフィルタし、選択変更時にAggregateをRefreshすることで、依存関係に応じた適切な値だけを動的に表示できます。全件ロードして非表示にする方法は非効率です。

Q20

外部データに対してユーザーの入力を検証し、保存する前に確認したい。どこにチェックを置くべき？

  • AJavaScript でチェックする
  • Bサーバーアクションを使い、エンティティの保存前に検証する
  • Cタイマーを使って保存時に検証する
  • D画面の送信ボタンだけで実行する
  • E保存後に警告を表示する
  • F静的エンティティの検証

正解：B

検証はServer Actionで行い、Entityに保存する前に外部データと照合します。クライアント側（JavaScript）だけの検証はバイパスされる可能性があるため、サーバー側でのチェックがデータ整合性の保証に不可欠です。
