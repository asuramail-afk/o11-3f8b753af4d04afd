Q1

モジュールが公開しているサーバーアクションを誤用されないようにするにはどうすればよいですか？

  • Aすべての呼び出しをログに記録する
  • Bロールチェックを行い、ロジック実行前に入力を検証する
  • CCSS で非表示にする
  • DJavaScript でチェックを行う
  • E匿名利用を許可する
  • F呼び出し元に信頼を置く

正解：B

公開ロジックは自己防衛が必要です。ロジック実行前にサーバー側でロール（Check<Role>）と入力データを検証し、不正な呼び出しから守ります。ログや UI/JS だけの対策では不十分です。

Q2

顧客のリクエストで、フィルターを適用したリストを Excel としてダウンロードさせるには？

  • A画面やロジックに組み込み済みの『Export to Excel』機能を使う
  • Bリストを表示してスクリーンショットを撮る
  • CJavaScript を使う
  • Dメールで送信するだけにする
  • EWeb API を作成する
  • FWeb ブロックを追加する

正解：A

OutSystems は標準の Excel エクスポート（Table/List の Export to Excel、または ListRecordsToExcel 系アクション）を提供しており、フィルタ済みのリストを迅速に出力できます。

Q3

各レコードの最終変更日時を追跡したい場合、最も適切な方法はどれですか？

  • Aタイマーを使って定期的に更新する
  • Bセッション変数を使って保持する
  • C追跡を無視して管理しない
  • DAggregateを使って取得する
  • ELastModified 属性を追加し、保存時にサーバーアクションで更新する
  • Fテーマに更新記録を追記する

正解：E

エンティティに LastModified（DateTime）属性を追加し、Create/Update を行うサーバーアクション内で CurrDateTime() を設定します。これで変更履歴を正確に追跡できます。

Q4

ローカル変数を機密データに使用した場合のリスクは何ですか？

  • Aエンティティ
  • BWeb ブロックの入力
  • Cテーマ変数（Theme vars）
  • DCheck<Role>() 関数
  • Eセッション変数
  • Fナビゲーション間では保持されず、デバッグ時に露出する可能性がある

正解：F

ローカル変数は一時的で画面遷移をまたいで保持されず、デバッグ時に値が露出し得ます。機密情報の保持には適しません（短時間・一時的なデータ向け）。

Q5

API が『次のページ』URL を含むページネーションされたレスポンスを返す場合、どう対応すればよいですか？

  • Aスタイルシートを使用する
  • Bページネーションをスキップする
  • Cサーバーアクションのループを使い、ページがなくなるまでページを取得する
  • Dレスポンスをハードコードする
  • E一度だけ送信する
  • FAggregateを使う

正解：C

サーバーアクション内でループし、レスポンスの『次のページ』URL がなくなるまで API を呼び続けて全ページを取得します（Aggregateは外部 API のページ送りには使えません）。

Q6

大量のデータを扱う画面のロードが遅い場合、パフォーマンス向上に役立つのは？

  • Aページング、インデックス作成、フィルターを Aggregate または SQL クエリに追加する
  • Bタイマーを使ってバッチでデータを実行する
  • Cすべてのレコードを最初にロードする
  • D古いデータを削除する
  • Eフィルターを省略する
  • FAPI から常に読み込む

正解：A

取得件数を制限（Max Records/ページング）し、インデックスとフィルターでクエリを絞ると、転送量とレンダリングが軽くなり高速化します。

データモデルの変更のデプロイが失敗しました。最初に何を確認すればよいですか？（複数）

  • A役割の確認（Check Roles）
  • Bスタイルシートの設定
  • Cプロセスのタイマー
  • Dエンティティの依存関係の検証
  • Eパブリッシュエラーの確認
  • F同期状態（Synchronization）の確認
  • Gテーマスタイル
  • Hデプロイのスキップ

正解：D・E・F

エンティティ変更は関係性やインデックスに影響します。エンティティの依存関係、パブリッシュエラー（TrueChange）、同期状態を確認して、スキーマ失敗の原因を特定します。

Q8

ビジネスアナリストがドロップダウンの値をいくつか更新したいが、開発ロジックは変えたくない。どうしますか？

※静的エンティティのレコードを編集した場合、反映にはモジュールの再公開が必要です。『再公開なしで実行時に値を更新』したい場合は、通常のエンティティ＋管理者用の編集画面（またはサイトプロパティ）を使います。本問の選択肢の中では静的エンティティが最も妥当ですが、この点に注意してください。

  • Aタイマーを追加
  • B式に値を追加
  • C新しいモジュールを作成
  • DService Studio で編集可能な静的エンティティを使用
  • Eセッション変数を使用
  • Fブロック内にハードコード

正解：D

静的エンティティは、ロジックを変えずに参照値（ドロップダウンの選択肢など）を一元管理・編集できます（ハードコードより保守的）。

Q9

リアルタイムの KPI ダッシュボードで動的なリフレッシュを行いたい。最適なアプローチは？

  • Aセッション変数を使ってデータを保存する
  • Bタイマーを使って画面全体をリロードする
  • C1つの KPI ずつ画面に表示する
  • DAggregateとリフレッシュトリガー（AJAX／クライアントイベント）を使う
  • EAPI から値をハードコードする
  • F静的ブロックにすべての KPI を表示する

正解：D

Aggregateを RefreshData（クライアント側イベント/タイマー）で部分更新すると、画面全体を再読み込みせずスムーズにリアルタイム表示できます。

Q10

複数のモジュールで共有されているエンティティを更新するとき、他の機能を壊さないために何をすべきですか？

  • Aモジュールを複製して変更
  • Bリグレッションテストを行い、パブリックな依存関係を注意深く更新する
  • Cエンティティを削除して再作成
  • D直接更新して公開
  • Eブロック内の JavaScript を更新
  • FTrueChange の警告を無視

正解：B

共有エンティティの変更は影響範囲（依存するモジュール）を理解した上で行い、リグレッションテストで他機能への波及を確認し、依存関係を慎重に更新・再公開します。

Q11

動的フィルターとページネーションを備えた商品カタログ画面で、効率的なデータ読み込みを実現する最良の方法は？

  • Aタイマーを使ってデータを取得する
  • BJavaScript を使ってフィルター処理を行う
  • CWeb ブロックを使ってデータを読み込む
  • D静的エンティティを使う
  • Eセッション変数にすべてのレコードを読み込む
  • FAggregateを使い、動的フィルターとページネーションを制御する

正解：F

Aggregateに動的フィルター（変数バインド）とページネーション（Max Records/Start Index）を組み合わせると、DB 側で絞り込み、高速かつ柔軟にカタログを表示できます。

Q12

監査のためにユーザーのアクティビティ（ページアクセスやデータ変更）をログに記録したい。推奨パターンは？

  • Aスタイルシートでログを記録する
  • Bタイマーを使う
  • Cロギングをスキップする
  • Dアクティビティ追跡を集中化するロギングコンポーネント／モジュールを作成する
  • Eセッション変数を使う
  • F画面ごとに手動でログを追加する

正解：D

ロギングは横断的関心事（cross-cutting concern）なので、専用のロギングモジュール/コンポーネントに集約すると、ビジネスロジックから分離でき再利用性・一貫性が高まります。

  • Aユーザー入力に頼る
  • BJavaScript で表示/非表示を制御する

Q15

外部 API が 500 や 403 のエラーを返す場合、アプリはどう対処すべきですか？

※消費（Consume）REST API のコールバックの正式名は 'OnAfterResponse'（送信前は 'OnBeforeRequest'）です。500/403 等の非 2xx は既定で例外が発生するため、例外ハンドラや OnAfterResponse でステータスコードを確認して処理します（原資料の『OnResponse』は OnAfterResponse を指します）。

  • Aタイマーを実行してリトライする
  • Bレスポンスのコールバックで例外を適切に処理する
  • C空のレスポンスを返す
  • Dエラーを無視してリトライする
  • E画面にのみエラーを表示する
  • Fユーザーに直接通知を送る

正解：B

レスポンスに基づくエラー処理を行い、耐障害性のある連携にします。

Q16

複数の検証に成功した後にのみ注文確認を送信するプロセスを、どうモデル化しますか？

  • AWeb ブロックにロジックを追加し、クライアント側から送信する
  • B検証をスキップし、先にメールを送る
  • C複数のサーバーアクションを例外処理とともにチェーンし、メール送信前に検証を行う
  • Dまずメールを送信し、その後に検証を行う
  • E画面上にアラートを表示する

正解：C

サーバーアクションを順にチェーンし、各段で例外処理を入れて検証を確実に行い、すべて成功した後にのみ確認メールを送信します（順序と整合性を保証）。

Q18

データソースが外部で、頻繁にアクセスできない場合の最適なユーザーエクスペリエンスのパターンは？

  • Aエラーでページを分断する

API 失敗時は、直近のキャッシュ値を『最終同期時刻』とともに表示し、リトライ手段を提供すると、透明性とフォールバックのある良い UX になります。
