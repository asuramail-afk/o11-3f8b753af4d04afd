Q1

稼働中の本番環境でデバッグモードが無効の状態で、フローの特定ポイントの変数値を確認したい。再デプロイせずに行う最も効果的な方法は？

  • Aブラウザのデベロッパーツールでサーバー側の変数を直接確認する。
  • BService Center の『一般』または『エラー』ログを確認する（コードに LogMessage を実装している場合）。
  • C本番から部分的な OML をダウンロードし、ローカル開発環境でデバッグする。
  • D本番の Service Center でデバッグモードを有効にしデバッガをアタッチする。

正解：B

本番ではデバッガをアタッチできません。事前に LogMessage で重要な変数を記録し、Service Center のログで確認するのが定石です。ブラウザツールはサーバー側変数を見られません。

Q2

外部 REST API 呼び出しで『Circuit Breaker（サーキットブレーカー）』エラーが出ました。何を意味しますか？

  • Aリクエストのペイロードサイズが最大許容限界を超えた。
  • B外部 API が繰り返し失敗し、OutSystems がリソース枯渇を防ぐため一時的にリクエスト送信を停止した。
  • C外部 API が 401 Unauthorized を返した。
  • D外部 API の SSL 証明書が期限切れになった。

正解：B

サーキットブレーカーは自動保護機構。タイムアウト等の失敗が閾値を超えると回路を『open』にし、以降の呼び出しを外部に送らず即失敗を返して、リソース枯渇やスレッドブロックを防ぎます。

Q3

高トラフィックの Logs テーブル（週100万件増）で、1年以上前のレコードを Logs_Archive に移動後に削除するアーカイブ戦略。最善は？

  • A外部ツールが全データを取得し削除命令を送る API を公開する。
  • B『Create』イベントで BPT を起動し、作成後1年待って移動する。
  • C単一 SQL の INSERT INTO ... SELECT の後に DELETE を実行する Timer を作る。
  • D毎日実行する Timer で古いレコードを限定バッチ（例1000件）ずつ Logs_Archive に挿入・Logs から削除し、各バッチ後にコミットする。

正解：D

大量データはタイマー内のバッチ処理が最適。バッチ単位でコミットすることで、タイムアウトやトランザクションログの圧迫・DB ロックを避けつつ安全に移動・削除できます。

Q4

タイマーで 50,000 件のレコードを処理します。トランザクションのタイムアウトエラーを避ける最適な方法は？

  • Aタイマーの Timeout プロパティを環境の最大値に設定する。
  • BUI の画面アクションを作り、REST API 経由でタイマーから呼ぶ。
  • C小さなバッチに分けて処理し、各バッチ後に明示的に CommitTransaction を呼んでタイムアウトをリセットするループを実装する。
  • D全 50,000 件を1つのサーバーアクションの For Each で処理し、コミットしない。

正解：C

バッチに分けて各バッチ後に CommitTransaction すると、リソースを解放しトランザクションのタイムアウトをリセットできます。Timeout 延長は一時しのぎで、データ増加で結局超過します。

Q5

次のうち、サービスアクション（Service Action）を使う正当なケースは？

  • A2つのアプリ間でロジックを共有しつつ、緩やかな結合と独立した展開ライフサイクルを維持したい場合。
  • BSAP などの外部システムにロジックを公開したい場合。
  • Cロジックを消費モジュールの DB トランザクション内で実行したい場合。
  • D数学的計算で可能な限り最高の実行速度が必要な場合。

正解：A

サービスアクションは内部的に REST ベースで動作する弱い依存。疎結合と独立展開を実現します。外部公開は公開 REST/SOAP、トランザクション参加や高速計算はサーバーアクション（インプロセス）が適します。

Q6

複数画面で使うブロック『ProductDetail』がAggregate『GetProduct』でデータを取得。『Load Details』ボタンが押されるまで GetProduct を自動実行しないようにするには？

  • AAggregateの Fetch を『At start』に設定する。
  • BAggregateを『If』ウィジェットの中に置く。
  • CAggregateの Fetch を『Only on demand』に設定する。
  • DAggregateを削除して『Data Action』を使う。

正解：C

Fetch=『Only on demand』にすると自動実行されず、明示的に RefreshAggregate を呼んだとき（例：ボタンクリック）だけ実行されます。

Q7

開発環境でタイマーのロジックにブレークポイントを設定しました。ブレークポイントをトリガーするには？

  • Ascheduled 時間を待てば自動的にデバッガが開く。
  • Bスケジューラーがトリガーするタイマー実行は直接デバッグできない。テスト画面のボタン等のフローを作り、その中でタイマーのアクションを明示的に呼ぶ。
  • CService Center の『Run Now』ボタンを使う。
  • D『Deployment Controller』サービスにデバッガをアタッチする。

正解：B

タイマーはバックグラウンドのシステムセッションで動くため、ユーザーセッションにアタッチするデバッガでは捕捉できません。テスト画面からタイマーのアクションを手動で呼び出してデバッグします。

Q8

パフォーマンスのため属性『CustomerName』にインデックスを追加します。この変更は Create/Update 操作にどう影響しますか？

  • ACreate/Update がわずかに遅くなる可能性がある。
  • Bテーブルが大きい場合、公開プロセス中にタイムアウトが発生する。
  • CCreate/Update に影響しない。
  • DCreate/Update が高速化される。

正解：A

インデックスは読み取り（SELECT）を高速化する一方、書き込み（INSERT/UPDATE/DELETE）ではインデックス構造の更新が必要なため、わずかに遅くなります（トレードオフ）。

Q9

Service Center の『Error』ログに『Request Transaction Aborted』が表示されました。最も一般的な原因は？

  • Aユーザーがリクエスト処理中にブラウザを閉じた。
  • Bリクエスト処理に10秒以上かかった。
  • Cユーザー例外または特定の DB 例外が発生し、適切に処理されずトランザクションがロールバックされた。
  • Dデータベースサーバーがオフライン。

正解：C

キャッチされない例外がトップレベルに達すると、プラットフォームはデータ整合性のためトランザクションを中断し、『Request Transaction Aborted』を記録します。タイムアウトは別の『Request Timed Out』エラーです。

Q10

モバイルアプリで、DOM が完全に読み込まれウィジェットが操作可能になったときだけロジックを実行したい。使うべきイベントは？

  • AOnRender
  • BOnParametersChanged
  • COnReady
  • DOnInitialize

正解：C

OnReady は画面/ブロックの DOM が準備完了し、全ウィジェットがレンダリングされ JavaScript からアクセス可能になったときに呼ばれます。OnInitialize は DOM レンダリング前です。

Q11

プロフィール画像を保存前に 200x200 にリサイズします。サーバー負荷を最小にするため、リサイズロジックはどこに実装すべき？

  • AExtension を使用したサーバーアクションにリサイズロジックを実装する（標準解）。
  • B『Image』ウィジェットの Width/Height プロパティで保存前にリサイズする。
  • C画像をそのまま保存し、表示時にだけリサイズする。
  • DJavaScript のクライアントアクションで、サーバー送信前にバイナリをリサイズする。

正解：A

OutSystems の標準機能に画像リサイズはなく、通常は Extension（Integration Studio の C# コード）や Forge コンポーネントでサーバー側処理します。Image ウィジェットのプロパティは表示のみに影響し、バイナリは変わりません。

Q12

公開する REST API のセキュリティのベストプラクティスは？

  • AAPI キーを認証のためロジック内にハードコードする。
  • B全メソッドに『匿名（Anonymous）』ロールを使い統合を容易にする。
  • C『Internal Access Only』を『No』に設定してセキュリティを確保する。
  • D認証（Basic Auth やカスタム OAuth 等）を適用し、SSL/TLS（HTTPS）で全公開エンドポイントを保護する。

正解：D

公開 API には適切な認証と HTTPS による通信暗号化が必須。API キーはハードコードせずサイトプロパティ/暗号化エンティティに保存し、Anonymous は機密操作に不適切です。

Q13

結果を10分キャッシュするサーバーアクション（Cache in Minutes = 10）をタイマーの中で呼び出すと何が起こりますか？

  • Aキャッシュ期間が自動的に倍になりバックグラウンド処理に適用される。
  • Bタイマーは失敗する（キャッシュされたアクションはバックグラウンドで非対応のため）。
  • Cキャッシングは無視され、そのロジックが毎回実行される。
  • Dアクションは最後のユーザーセッションのキャッシュ値を返す。

正解：C

タイマーは実行ごとに独立したシステムセッションで動くため、ユーザーセッションと共有されるキャッシュは跨って再利用されず、実質的にロジックが毎回実行されます。跨る再利用が必要なら分散キャッシュを使います。

Q14

リアクティブ Web アプリの Data Action で、『Employees』と『Departments』の2つのリストを返す必要があります。どう実現しますか？

  • AData Action に2つの出力パラメータ（Employees リストと Departments リスト）を追加し、フロー内で割り当てる。
  • B1つの Data Action は1リストしか返せないため、2つの別々の Data Action を作る。
  • C両リストを含むStructureを作り、それを唯一の出力パラメータにする。
  • DData Action はリストを返せず基本型のみ。

正解：A

Aggregateと異なり、Data Action は複数の出力パラメータを持て、各パラメータは任意のデータ型（リスト含む）にできます。Structureで1出力にするのも可能ですが、複数出力の方が直接的です。

Q15

エンティティ『Order』と『OrderLine』。Order を削除したとき関連する全 OrderLine も自動削除し孤児を残さないようにするには、OrderLine.OrderId の Delete Rule をどう設定？

  • ANullify（null に設定）
  • BProtect（保護）
  • CDelete（削除）
  • DIgnore（無視）

正解：C

Delete ルールは親（Order）削除時に子（OrderLine）も自動削除するカスケード削除を作り、孤児レコードを防ぎます。Protect は関連子があると親削除を禁止、Nullify は FK を NULL 化します。

Q16

サードパーティ REST API 利用時、各リクエストにカスタム認証トークンを HTTP ヘッダーに含めたい。どのコールバックを使う？

  • AOnBeforeRequest
  • BOnRequestHeader
  • COnAfterResponse
  • DOnAuthentication

正解：A

OnBeforeRequest は、リクエストを外部に送信する前に Request オブジェクト（ヘッダー・URL・ボディ）を操作できるため、ここで認証トークンを追加します。OnAfterResponse は受信後なので遅すぎます。

Q17

リアクティブ Web アプリの画面で Aggregate を使ってデータ取得。データが取得され利用可能になった直後に特定のクライアントアクションを実行したい。どうする？

  • A『OnReady』で Aggregate リストが空でないか確認する。
  • B画面の『OnRender』でクライアントアクションを呼ぶ。
  • C『OnInitialize』の最後でクライアントアクションを呼ぶ。
  • DAggregate の『On After Fetch』プロパティに目的のクライアントアクションを設定する。

正解：D

Aggregate は非同期に動作します。取得完了と連携するには、その Aggregate の『On After Fetch』イベントハンドラを使うのが最適。OnReady/OnInitialize は取得前に走る可能性があります。

Q18

古いレコードをアーカイブする夜間バッチが必要で、完了に約2時間かかります。最も適した仕組みは？

  • A外部スケジューラーから呼ばれる REST API を公開する。
  • Bスケジュール設定され、コミット/タイムアウトを処理するロジックを持つタイマー。
  • Cホームページの JavaScript『setInterval』でトリガーされる画面アクション。
  • D『Wait』アクティビティを含む BPT プロセス。

正解：B

Timer は定期バックグラウンドバッチの標準ツール。2時間の長時間実行でも、途中で CommitTransaction 等のチェックポイントを入れて実行タイムアウトを避ければ実現できます。

Q19

リアクティブ Web アプリのブロックに入力パラメータ『EntityId』があります。親画面が EntityId を更新するたびに DB から新データを取得したい。どのライフサイクルイベント？

  • AOnReady
  • BOnRender
  • COnInitialize
  • DOnParametersChanged

正解：D

OnParametersChanged は入力パラメータの値が変わったときにのみ呼ばれるため、親から新しい EntityId が渡されたときのデータ再取得に最適です。OnReady/OnInitialize は初回一度きりです。

Q20

REST API メソッドが JSON ペイロードを受け取ります。クライアントが特定の無効値を送った場合に、400 Bad Request とカスタムエラーメッセージを返すには？

  • A標準の DB 例外をスローすると自動的に 400 Bad Request に翻訳される。
  • B公開 REST API のステータスコードはカスタマイズできない。
  • C『Assign』で出力パラメータの Status を 400 に設定しフローを終了する。
  • DAPI フロー内で入力を検証し、不正なら HTTPRequestHandler 拡張の SetStatusCode で 400 を設定し、カスタムメッセージとともに User Exception をスローする。

正解：D

入力を検証し、不正時は HTTPRequestHandler の SetStatusCode でステータスを 400 にしてから User Exception をスローすると、カスタムエラーメッセージ付きで 400 を返せます。
