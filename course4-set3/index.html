Q1

Projects と Tasks（1:N）があります。各プロジェクトとそのタスク数を表示したい。最適な方法は？

  • A静的エンティティにカウントを追加する
  • BSQL クエリを手動で実行する
  • Cセッション変数にカウントを保存する
  • Dグループ化とカウントを使った Aggregate を使用（ProjectId でグループ化し、Task をカウント）
  • Eプロジェクトごとにタイマーを設定
  • Fタスクリストを表示してから JavaScript でカウントする

正解：D

Aggregate で ProjectId によるグループ化と Count を使えば、各プロジェクトのタスク数を1つのクエリで効率的に取得できます。JavaScript での集計や Timer は不要で、データベース側で計算されるためパフォーマンスにも優れます。

Q2

レポートで複数の倉庫のリアルタイム在庫情報を表示します。正確性をどう確保しますか？

  • Aセッションにキャッシュ
  • Bタイマーで定期的に更新
  • C静的リストを表示
  • Dデータを一度プリロード
  • EAPI から条件付きのリフレッシュを行うサーバーアクションを使用
  • Fローカル変数を使う

正解：E

必要なタイミングで Server Action から最新の在庫データをオンデマンドに取得すれば、リアルタイムの正確性を確保できます。Session へのキャッシュやプリロードでは、表示中にデータが古くなってしまいます。

Q3

アプリ全体でダークモードのトグルを有効にする、最もスケーラブルな方法は？

  • Aセッション変数と条件付きスタイルを使ったテーマ切り替え
  • BJavaScript だけでトグルを切り替え
  • Cエンティティを使う
  • Dカスタム CSS をインラインで追加
  • EWeb ブロックの背景のみを変更
  • F各画面ごとに変更

正解：A

テーマ設定を Session Variable に保持し、条件付きスタイルでクラスを切り替えれば、アプリ全体で一元的にダークモードを適用でき、最もスケーラブルです。画面ごとの個別変更やインライン CSS では保守性が損なわれます。

Q4

ユーザーが5分間操作しない場合に警告を表示したい。どうしますか？

  • A常に画面に表示する
  • Bクライアントサイドの JavaScript を使い、セッションタイムアウトの警告モーダルを表示する
  • Cタイマーで確認する
  • Dページ全体をリフレッシュする
  • Eアイドル動作を無視する
  • Fサーバーサイドのロジックだけを使う

正解：B

無操作（アイドル）状態の検知はブラウザ上でしかできないため、クライアントサイドの JavaScript で操作を監視し、5分経過したら警告モーダルを表示します。Timer はサーバー側のバックグラウンド処理であり、ユーザーの操作を検知できません。

Q5

チームメンバーがローカルでは動作していたロジックが、デプロイ後に壊れました。考えられる原因は？

  • Aスタイルシートのみを使用した
  • Bクライアントサイドの変数や非公開（Private）のアクションをモジュール間で使おうとした
  • Cエンティティを公開設定にした
  • DCSS クラスを使用した
  • Eタイマーを二度実行した
  • Fブロックを誤って表示した

正解：B

モジュール間で要素を参照するには、その要素が Public に設定されている必要があります。非公開（Private）のアクションやクライアント側のスコープに閉じた変数に依存していると、参照が解決できずデプロイ後に動作が壊れます。

Q6

Web API でエンティティから選択したフィールドだけを公開したい。最適な方法は？

  • AJSON ですべての値を表示する
  • Bデータベース ID を公開する
  • C必要なフィールドだけをマッピングした出力 Structure（DTO）にする
  • DWeb ブロックを追加する
  • E完全なエンティティを返す
  • Fタイマーを使ってブロックする

正解：C

必要な属性だけを持つ出力用の Structure（DTO）を定義してマッピングすれば、公開するフィールドを明示的に制御でき、機密情報の漏洩を防げます。Entity をそのまま返すと不要な内部情報まで公開してしまいます。

Q7

必須項目が空の場合、ユーザーがフォームの保存をできなくする最も良い方法は？

  • A常にボタンを無効にする
  • Bビルトインのバリデーションと Form.Valid 組み込み式を使用する
  • CJavaScript のチェックを追加する
  • Dタイマーによるチェック
  • E保存後に警告を表示する
  • F非表示フィールドを使用する

正解：B

入力の Mandatory 設定などビルトインのバリデーションと Form.Valid 組み込み式を組み合わせ、保存前に検証結果を確認して無効なら処理を止めるのが標準的な方法です。独自の JavaScript チェックや保存後の警告は不要な複雑さや手遅れの対応になります。

Q8

顧客レコードをエクスポートし、計算された割引を含めたい。ロジックはどこに定義すべき？

  • AAggregateだけ
  • Bセッション変数
  • C送信後に表示
  • Dタイマー
  • EExcel にエクスポートする前に使用するサーバーアクション内
  • FUI 式

正解：E

割引などの計算ロジックは Excel エクスポートの前段となる Server Action 内に定義し、出力用データを整えてからエクスポートします。Aggregate だけでは複雑な行単位の計算ロジックを表現しきれず、UI 式ではエクスポートに反映されません。

承認ワークフローを Admin と Manager で異なる分岐にしたい。どう実装しますか？

  • AWeb ブロックを使用する
  • B先にメールを送る
  • CUI だけのロジックを使用する
  • Dタイマーを使用した分岐
  • Eプロセス/サーバーロジック内で Check<Role>() と If-Else を使用する
  • Fフロントエンドに依存する

正解：E

サーバー側のロジックで Check<Role>() を評価し、If で Admin と Manager の分岐を作るのが安全な実装です。ロール判定を UI やフロントエンドだけに頼ると、改ざんされる恐れがありセキュリティを担保できません。

大きなAggregateデータのために画面の読み込みが遅い。最も効率的な解決策は？

  • A上位10行だけハードコーディング
  • Bタイマーを実行して遅延させる
  • Cセッション変数を使用してキャッシュ

Aggregate に Max Records によるページネーションとフィルターを直接設定すれば、データベース側で取得件数が絞られ、クエリ実行と画面描画の両方が効率化されます。全件取得してからキャッシュや JavaScript で処理する方法は根本解決になりません。

  • BJavaScript でロック/制御する
  • C編集可能なフィールドを表示し、警告を出す

  • Eセッション変数のフラグだけを設定する
  • F編集を許可し、保存時にだけ止める

正解：D

フィールドの読み取り専用プロパティと、ロールチェックによる編集ボタンの非表示を組み合わせ、UI とロジックの両面で編集を防ぐのが適切です。JavaScript や Session Variable のフラグだけの制御は回避される可能性があります。

Q12

2つのモジュールが同じ User エンティティを使い、片方の更新がもう一方にエラーを起こしました。防ぐには？

  • A両方を一緒に更新する
  • Bセッションデータを使用する
  • Cパブリック要素とガバナンスルールを持つインターフェース（サービス）モジュールを使用する
  • Dエラーをスキップする
  • Eすべてのモジュールの結合を解除する
  • F依存関係を無視する

正解：C

共有する Entity やロジックは Public 要素を持つインターフェース（サービス）モジュールに切り出し、ガバナンスルールのもとで一元管理します。これにより変更の影響範囲が明確になり、片方の更新が他方を壊す事態を防げます。

Q13

リスト画面で1行に3つの関連値を表示する最適な方法は？

  • Aタイマーを追加
  • B1つの値だけを画面に表示
  • Cデフォルトのフィールドだけを表示
  • DWeb ブロックをネストする
  • Eクライアント側で JavaScript を使用
  • F必要なすべてのフィールドを含むエンティティ結合（Join）を使ったAggregate

正解：F

Aggregate で関連する Entity を Join し、必要な属性をまとめて取得するのが最適です。1回のクエリで1行に表示する複数の関連値がそろうため、Web Block のネストや JavaScript での加工より効率的でシンプルです。

Q14

REST API の呼び出しが頻繁にトークンの有効期限切れで失敗します。スケーラブルに解決するには？

  • Aロールを確認する
  • B常にリトライする
  • Cトークンをハードコーディングする
  • Dリフレッシュトークンのロジックを使用し、トークンをサイトプロパティまたはセッション変数に安全に保存する
  • Eエラーのみを表示する
  • F常に新しいリクエストを送る

正解：D

リフレッシュトークンのロジックで期限切れ時にトークンを自動更新し、Site Property や Session Variable に安全に保存して再利用するのがスケーラブルな解決策です。ハードコーディングや単純なリトライでは期限切れの根本原因に対処できません。

100万件規模の移行は、バッチ処理を行う Timer で分割して取り込み、エラー処理と再実行の仕組みを組み込むのが最適です。これによりタイムアウトを回避しつつ、制御された安全なマイグレーションができます。

  • AWeb ブロックの UI ロジックのみを使用
  • BTimer を追加してチェック
  • Cすべてをロードし、クライアント側（JS）でフィルターをかける
  • Dすべてのデータを表示し、非表示にする
  • EAggregate をマネージャーのチーム ID（セッション変数から取得）でフィルタリングする

Q18

すべてのログイン失敗を記録する必要があります。どこに追加しますか？

  • Aアラートだけを表示
  • Bタイマー後に実行
  • Cログインロジック内のサーバーアクションに、失敗時に条件付きで記録する
  • DCSS アラートを使用
  • EJavaScript で記録
  • Fロールを追加

正解：C

ログイン処理を行う Server Action 内で、認証失敗時に条件付きでログを記録すれば、永続的で信頼できる監査証跡になります。JavaScript などクライアント側での記録は改ざん可能で、永続性も保証されません。

Q19

  • A環境ごとにメールを送信
  • Bタイマーを使ってブロック

  • EJavaScript で URL をチェック

正解：F

Site Property は環境ごとに実効値を変更できるため、環境固有の設定値とアクセス制御ロジックを組み合わせれば、ステージングと本番でデータアクセスを確実に切り替えられます。JavaScript で URL を判定する方法は信頼性がなく安全ではありません。

Q20

API に認証ヘッダーが必要で、その内容を実行時に更新しなければなりません。最も良い方法は？

  • Aアクション内にハードコードする
  • Bヘッダーにロールを追加する
  • Cログイン情報を画面に表示する
  • Dサイトプロパティやセッション変数を REST ヘッダーにマッピングして使用する
  • E静的なヘッダーを追加する
  • Fエンティティ属性を追加する

正解：D

認証トークンなどの値を Site Property や Session Variable に保持し、REST 呼び出し時（OnBeforeRequest など）にヘッダーへ動的にマッピングするのが最適です。ハードコードを避けられ、実行時に値を安全に更新できます。
