Q1

Projects と Tasks（1:N）があります。各プロジェクトとそのタスク数を表示したい。最適な方法は？

  • A静的エンティティにカウントを追加する
  • BSQL クエリを手動で実行する
  • Cセッション変数にカウントを保存する
  • Dグループ化とカウントを使った Aggregate を使用（ProjectId でグループ化し、Task をカウント）
  • Eプロジェクトごとにタイマーを設定
  • Fタスクリストを表示してから JavaScript でカウントする

正解：D

Aggregate で ProjectId ごとにグループ化し Count することで、レコード単位のタスク数を単一クエリで効率的に取得できます。

Q2

レポートで複数の倉庫のリアルタイム在庫情報を表示します。正確性をどう確保しますか？

  • Aセッションにキャッシュ
  • Bタイマーで定期的に更新
  • C静的リストを表示
  • Dデータを一度プリロード
  • EAPI から条件付きのリフレッシュを行うサーバーアクションを使用
  • Fローカル変数を使う

正解：E

必要なときにサーバーアクションでバックエンドの最新状態をオンデマンド取得すれば、正確なリアルタイム在庫を表示できます（キャッシュやプリロードはすぐ古くなります）。

Q3

アプリ全体でダークモードのトグルを有効にする、最もスケーラブルな方法は？

  • Aセッション変数と条件付きスタイルを使ったテーマ切り替え
  • BJavaScript だけでトグルを切り替え
  • Cエンティティを使う
  • Dカスタム CSS をインラインで追加
  • EWeb ブロックの背景のみを変更
  • F各画面ごとに変更

正解：A

ユーザー設定をセッション変数に保持し、テーマ/条件付きスタイルクラスで切り替えると、中央制御でグローバルにスケーラブルにダークモードを適用できます。

Q4

ユーザーが5分間操作しない場合に警告を表示したい。どうしますか？

  • A常に画面に表示する
  • Bクライアントサイドの JavaScript を使い、セッションタイムアウトの警告モーダルを表示する
  • Cタイマーで確認する
  • Dページ全体をリフレッシュする
  • Eアイドル動作を無視する
  • Fサーバーサイドのロジックだけを使う

正解：B

クライアント側 JavaScript でアイドル（無操作）を検知し、警告モーダルを表示します（タイマーはサーバー側の背景処理でユーザーの操作は検知できません）。

Q5

チームメンバーがローカルでは動作していたロジックが、デプロイ後に壊れました。考えられる原因は？

  • Aスタイルシートのみを使用した
  • Bクライアントサイドの変数や非公開（Private）のアクションをモジュール間で使おうとした
  • Cエンティティを公開設定にした
  • DCSS クラスを使用した
  • Eタイマーを二度実行した
  • Fブロックを誤って表示した

正解：B

モジュール間で参照するには要素が Public である必要があります。非公開（Private）要素やスコープの合わない変数に依存すると、参照が解決できずデプロイ後に壊れます。

Q6

Web API でエンティティから選択したフィールドだけを公開したい。最適な方法は？

  • AJSON ですべての値を表示する
  • Bデータベース ID を公開する
  • C必要なフィールドだけをマッピングした出力 Structure（DTO）にする
  • DWeb ブロックを追加する
  • E完全なエンティティを返す
  • Fタイマーを使ってブロックする

正解：C

必要な属性だけを持つ出力 Structure（DTO：Data Transfer Object）を定義して返すと、機密情報の漏洩を防ぎ、安全で簡潔な API になります。

Q7

必須項目が空の場合、ユーザーがフォームの保存をできなくする最も良い方法は？

  • A常にボタンを無効にする
  • Bビルトインのバリデーションと Form.Valid 組み込み式を使用する
  • CJavaScript のチェックを追加する
  • Dタイマーによるチェック
  • E保存後に警告を表示する
  • F非表示フィールドを使用する

正解：B

OutSystems のネイティブなフォーム検証（必須フィールド）と Form.Valid 式を使い、保存前に入力状態を確認して無効時は保存を止めます。

Q8

顧客レコードをエクスポートし、計算された割引を含めたい。ロジックはどこに定義すべき？

  • AAggregateだけ
  • Bセッション変数
  • C送信後に表示
  • Dタイマー
  • EExcel にエクスポートする前に使用するサーバーアクション内
  • FUI 式

正解：E

割引などの計算ロジックはサーバーアクションに集約し、エクスポート用データを準備してから Excel 出力します（Aggregateだけでは行単位の計算ロジックを持てません）。

Q9

承認ワークフローを Admin と Manager で異なる分岐にしたい。どう実装しますか？

  • AWeb ブロックを使用する
  • B先にメールを送る
  • CUI だけのロジックを使用する
  • Dタイマーを使用した分岐
  • Eプロセス/サーバーロジック内で Check<Role>() と If-Else を使用する

サーバー側のロジックで Check<Role>() を評価し、If-Else でロールに応じたルートに分岐させます（UI やフロントエンド依存は安全でありません）。

Q10

大きなAggregateデータのために画面の読み込みが遅い。最も効率的な解決策は？

  • A上位10行だけハードコーディング
  • Bタイマーを実行して遅延させる
  • Cセッション変数を使用してキャッシュ
  • DAggregateにページネーションとフィルターを直接追加する
  • Eデータを JavaScript に送信
  • FWeb ブロック内にロード

正解：D

Aggregateに Max Records（ページネーション）とフィルターを直接加えて、DB 側で取得件数を絞ると、クエリ実行とレンダリングが最適化されます。

Q11

ユーザーにレコードの閲覧は許可するが、編集はできないようにしたい。どう実装しますか？

  • Aタイマーを使ってレコードをロックし制御する
  • BJavaScript でロック/制御する
  • C編集可能なフィールドを表示し、警告を出す
  • D読み取り専用フィールド（プロパティ）とロールチェックを使って編集ボタンを隠す
  • Eセッション変数のフラグだけを設定する
  • F編集を許可し、保存時にだけ止める

正解：D

フィールドの読み取り専用プロパティと Check<Role>() による編集ボタンの制御を組み合わせ、UI とロジックの両方で編集を防ぎます（UI 非表示だけ・JS だけは回避され得ます）。

Q12

2つのモジュールが同じ User エンティティを使い、片方の更新がもう一方にエラーを起こしました。防ぐには？

  • A両方を一緒に更新する
  • Bセッションデータを使用する
  • Cパブリック要素とガバナンスルールを持つインターフェース（サービス）モジュールを使用する
  • Dエラーをスキップする
  • Eすべてのモジュールの結合を解除する
  • F依存関係を無視する

正解：C

共有ロジック/データはインターフェース（サービス）モジュールに切り出し、公開要素とガバナンスルールで一元管理すると、再利用性を保ちつつ変更の波及を制御できます。

リスト画面で1行に3つの関連値を表示する最適な方法は？

  • Aタイマーを追加
  • B1つの値だけを画面に表示
  • Cデフォルトのフィールドだけを表示
  • DWeb ブロックをネストする
  • Eクライアント側で JavaScript を使用
  • F必要なすべてのフィールドを含むエンティティ結合（Join）を使ったAggregate

Aggregate で必要なエンティティを Join し、必要な属性をまとめて取得すると、1行に複数の関連値を効率的かつシンプルに表示できます。

Q14

REST API の呼び出しが頻繁にトークンの有効期限切れで失敗します。スケーラブルに解決するには？

  • Aロールを確認する
  • B常にリトライする
  • Cトークンをハードコーディングする
  • Dリフレッシュトークンのロジックを使用し、トークンをサイトプロパティまたはセッション変数に安全に保存する
  • Eエラーのみを表示する
  • F常に新しいリクエストを送る

正解：D

リフレッシュトークンで期限切れ時に自動更新し、トークンをサイトプロパティ/セッション変数に安全に保存して再利用すれば、認証を維持できます（ハードコードや単純リトライは非対応）。

Q15

外部システムから100万件のレコードをマイグレーションします。最適な戦略は？

  • A画面にエラーを表示
  • Bレコードごとに API を実行
  • Cすべてをメモリ内で処理
  • Dバッチのタイマーを使用し、統合とエラー処理を行う
  • E画面経由で送信
  • F一度限りのスクリプトを使用

正解：D

大量移行はバッチ処理のタイマーで、区切って取り込み・コミットし、エラー処理と再試行を組み込むと、タイムアウトを避けつつ制御された移行ができます。

Q16

アプリで 10,000 行の CSV をアップロードして処理します。どう対応しますか？

  • AJavaScript ループを実行する
  • B都度手動で送信する
  • C1行ずつ画面ごとに処理する
  • Dタイマーや非同期ロジックを使ってバックグラウンドで処理する
  • Eファイル全体をメモリに読み込む
  • F画面内のロジックで処理する

正解：D

大量行の処理はタイマー/非同期ロジックでバックグラウンド実行すると、UI の応答性を保ちタイムアウトを防げます（画面内での同期処理は遅延/タイムアウトの原因）。

Q17

マネージャーは自分のチームが送信したレコードのみを閲覧する必要があります。Aggregate でどう実現しますか？

  • AWeb ブロックの UI ロジックのみを使用
  • BTimer を追加してチェック
  • Cすべてをロードし、クライアント側（JS）でフィルターをかける
  • Dすべてのデータを表示し、非表示にする
  • EAggregate をマネージャーのチーム ID（セッション変数から取得）でフィルタリングする
  • Fテーマを使用

クエリ（Aggregate）レベルで、セッションのチーム ID を条件にフィルタすると、安全でスケーラブルにデータを絞り込み、権限外データの漏洩を防げます。

Q18

すべてのログイン失敗を記録する必要があります。どこに追加しますか？

  • Aアラートだけを表示
  • Bタイマー後に実行
  • Cログインロジック内のサーバーアクションに、失敗時に条件付きで記録する
  • DCSS アラートを使用
  • EJavaScript で記録
  • Fロールを追加

ログイン処理のサーバーアクション内で、認証失敗時に条件付きでエンティティ/ログに記録すれば、永続的で安全な監査証跡になります（クライアント側記録は改ざん・非永続の問題）。

Q19

ステージング環境と本番環境のデータへのアクセスを制限するには？

  • A環境ごとにメールを送信
  • Bタイマーを使ってブロック
  • Cすべてのデータを表示
  • Dテーマを使って非表示にする
  • EJavaScript で URL をチェック
  • F環境固有のサイトプロパティとアクセスロジックを使用する

正解：F

サイトプロパティは環境ごとに実効値を変えられます。環境固有の設定＋アクセスロジックで、環境ごとのデータアクセスを確実に制御できます（クライアント側チェックは信頼性が低い）。

Q20

API に認証ヘッダーが必要で、その内容を実行時に更新しなければなりません。最も良い方法は？

  • Aアクション内にハードコードする
  • Bヘッダーにロールを追加する
  • Cログイン情報を画面に表示する
  • Dサイトプロパティやセッション変数を REST ヘッダーにマッピングして使用する
  • E静的なヘッダーを追加する
  • Fエンティティ属性を追加する

正解：D

認証トークン等の値をサイトプロパティ/セッション変数に持ち、REST の（OnBeforeRequest 等で）ヘッダーに動的にマッピングすると、ハードコードを避け実行時に安全に更新できます。
