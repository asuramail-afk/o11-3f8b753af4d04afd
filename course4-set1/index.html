Q1

エンティティに Is_Deleted 属性がある場合、何が可能になりますか？

  • AUI がクラッシュする
  • Bエンティティが削除される
  • Cアクションが中断される
  • Dソフトデリート（論理削除）をサポートする
  • Eタイマーが動作する
  • Fセッションが終了する

正解：D

Is_Deleted のようなフラグ属性を持たせると、レコードを物理的に削除せず「削除済み」として論理的に扱うソフトデリート（論理削除）が可能になります。データを保持したまま画面やクエリから除外できるのが利点です。

Q2

Aggregates を SQL クエリより使うメリットは何ですか？

  • Aトリガーをタイマーに設定する
  • Bより高速
  • CCSS のために必要
  • Dロールを追加する
  • E常に必要
  • F使いやすく、視覚的で、安全（ビルトインの最適化とともに使用できる）

正解：F

Aggregate は視覚的に作成できて習得が容易なうえ、SQL インジェクションを防ぐ安全性と、実際に使用する属性だけを取得するビルトインの最適化を備えています。このため通常のデータ取得では SQL クエリより Aggregate が推奨されます。

Q3

アクションの『例外ハンドラー』の目的は何ですか？

  • Aデータを保存する
  • Bセッションをトリガーする
  • CJavaScript を追加する
  • Dエラーをキャッチして管理し、アプリの失敗を避ける
  • Eタイマーを実行する
  • Fスタイルを定義する

正解：D

Exception Handler は、実行中に発生したロジックやデータベースのエラーをキャッチして処理し、アプリケーションが異常終了するのを防ぎます。エラーメッセージの表示やロールバックなど、失敗時の挙動を制御できます。

Q4

Web ブロックにデータを渡す正しい方法は何ですか？

  • Aセッション変数を使う
  • Bロールを使う
  • CJavaScript を使う
  • DAggregateを使う
  • EWeb ブロック内で定義した入力パラメータを使う
  • Fエンティティを使う

正解：E

Web Block は再利用可能なコンポーネントであり、外部からデータを受け取る正しい方法は Web Block に定義した入力パラメータです。Session Variable や JavaScript で渡すのは適切ではありません。

Q5

外部データベースに接続するにはどうすればいいですか？

  • Aテーマクラス
  • BWeb API
  • CCDT（Client Data Table）
  • Dエンティティ
  • Eプロセスタイマー
  • F外部データベース接続を作成し、エンティティを定義する

正解：F

外部データベースとの連携は、外部データベース接続を作成し（Integration Studio や Integration Builder を使用）、対象テーブルを Entity として定義することで実現します。定義した Entity は通常の Entity と同様に Aggregate などから利用できます。

Q6

Web Block を複数の画面で再利用すると、何が起こりますか？

  • Aテーマが変更される
  • B変数がリセットされる
  • C同じロジックと UI が一貫して再利用される
  • Dタイマーがトリガーされる
  • E壊れる
  • Fデータが保存される

正解：C

Web Block を複数の画面で再利用すると、同じ UI とロジックが一貫して適用されます。修正も 1 か所で済むため、重複を排除（DRY）しつつ保守性と一貫性を高められます。

Q7

画面間でレイアウトの一貫性を確保するにはどうすればよいですか？

  • Aアクション
  • B共有レイアウトモジュール／テンプレートの使用
  • Cタイマー
  • Dグリッド繰り返し
  • Eプロセス
  • Fエンティティ

正解：B

共有レイアウト（レイアウトテンプレートやレイアウト用 Module）を使うと、すべての画面に同じ構造とビジュアルを適用でき、レイアウトの一貫性を確保できます。

Q8

Aggregate と SQL クエリの主な違いは何ですか？

  • Aセッション変数
  • Bテーマ
  • Cエンティティ
  • Dアクション
  • Eプロセス
  • FAggregate は視覚的で安全、SQL は完全な制御を提供する

正解：F

Aggregate は視覚的で安全かつ自動最適化が働くため、通常のデータ取得に適しています。一方 SQL クエリは、複雑な結合や高度な条件など、クエリを完全に制御したい場合に使用します。

Q9

クライアントアクションとサーバーアクションの違いは何ですか？

  • Aフロントエンドのみ
  • Bタイマー
  • CCDT（自己完結型データ型）
  • Dロジックではない
  • Eクエリのみ
  • Fクライアントはデバイス／ブラウザ上で実行され、サーバーはバックエンドで実行される

正解：F

Client Action はユーザーのデバイスやブラウザ上で実行され、即時の応答性が求められる UI ロジックに適しています。Server Action はサーバー側で実行され、ビジネスロジックやデータベースアクセスを担当します。

Q10

画面からサーバーアクションに値を渡すにはどうすればよいですか？

  • Aテーマフィールド
  • Bセッション変数
  • Cタイマー
  • DCDT Structure
  • E入力パラメータ（単独）
  • F入力値をサーバーアクションのパラメータにバインドする

正解：F

画面からの値は、Server Action に定義した入力パラメータに画面の入力値をバインドして渡します。Session Variable などを経由させる必要はありません。

Q11

ライブアプリケーションのエラーを追跡するのに役立つ機能は何ですか？

  • Aテーマ設定
  • Bタイマーモニター
  • Cウェブスクリーンプレビュー
  • Dエンティティリスト
  • Eプロセスフロー
  • Fサービスセンターのログ

正解：F

Service Center のログ（Error ログや General ログなど）を使うと、稼働中のアプリケーションで発生した例外やエラーをほぼリアルタイムに確認・追跡できます。

Q12

多対多のリレーションシップはどのようにモデル化しますか？

  • Aテーマを使用して
  • B2つの外部キーを持つ中間（ジャンクション）エンティティを作成して
  • CAggregateを使用して
  • Dセッション変数を使用して
  • Eタイマーを使って
  • Fロールを使って

正解：B

多対多のリレーションシップは、両方の Entity への外部キーを持つ中間（ジャンクション）Entity を作成してモデル化します。この中間 Entity が 2 つの Entity の組み合わせを 1 レコードとして表現します。

Q13

どのアクションタイプがサーバー側で実行されますか？

  • AJavaScript
  • Bエンティティ
  • Cサーバーアクション
  • DWeb スクリーン
  • Eプロセストリガー（タイマー）
  • Fクライアントアクション

正解：C

サーバー側で実行されるのは Server Action です。ビジネスロジックやデータベース操作をサーバー上でカプセル化して実行します。Client Action はブラウザやデバイス側で実行されます。

Q14

『Ajax Refresh』の用途は何ですか？

  • Aロールを更新する
  • BUI をリロードする
  • Cタイマーを更新する
  • Dアクション
  • E画面全体をリフレッシュせずに部分的に UI を更新する

正解：E

Ajax Refresh（従来の Web アプリケーション）は、画面全体を再読み込みせずに対象の UI 部分だけを更新します。これにより、応答性が高くシームレスなユーザー体験を実現できます。

Q15

サイトプロパティの一般的な用途は何ですか？

  • Aプロセス入力
  • Bテーマスタイル

  • Dタイマー設定
  • Eエンティティ属性
  • Fセッション変数

正解：C

Site Property は、定数やアプリケーション全体で共有する設定値を格納するためのグローバルな仕組みです。値は Service Center から環境ごとに変更でき、再発行なしで設定を切り替えられます。

Q16

モジュールを発行（Publish）すると何が起こりますか？

  • Aタイマーが再起動される
  • Bデータベースがクリアされる
  • Cすべてのロールがリセットされる
  • Dモジュールとそのすべての変更がコンパイルされてデプロイされる
  • Eユーザーがログアウトされる
  • Fデータが削除される

正解：D

Module を発行（Publish）すると、そのすべての変更がコンパイルされて環境にデプロイされ、最新のロジックやデータモデルが有効になります。データベースのデータが消えたりユーザーがログアウトされたりすることはありません。

Q17

変数（や要素）を『Public』としてマークすることの利点は何ですか？

  • A自動実行する
  • Bメールをトリガーする
  • CJavaScript を有効にする
  • DRole を追加する
  • ECSS で使用する
  • F他のモジュールやコンポーネントからアクセスできる

正解：F

要素を Public にすると、同じ環境（ファクトリ）内の他の Module から Reference として参照・再利用できるようになります。Public でない要素は、その Module の内部でのみ使用できます。

Q18

複数のアプリ間でロジックを共有できる機能は何ですか？

  • AAggregate
  • Bスタイル（Styles）
  • CWeb ブロック（Web Blocks）
  • Dパブリック要素を持つモジュール（Modules with public elements）
  • Eセッション変数（Session variables）
  • FCDT（カスタムデータ型）

正解：D

複数のアプリケーション間でロジックを共有するには、Public 要素を持つ Module を利用します。他の Module がそれを Reference として参照することで、Server Action などのロジックを再利用できます。Web Block は主に UI の再利用、Structure はデータ型の定義が目的です。

式（Expression）は、計算結果や変数の値などの動的なコンテンツを UI 上に表示するために使います。画面上で値を計算・整形して見せるのが主な役割です。

Q23

入力項目を『必須』に設定した場合の効果は何ですか？

  • C自動的に保存される

入力項目を必須（Mandatory）に設定すると、ユーザーは送信前に必ず値を入力しなければならなくなります。ビルトインのバリデーションが未入力の送信を防ぎます。

Q25

OutSystems には入力検証の仕組みが標準で組み込まれています。必須フィールドの設定や、Valid / ValidationMessage を使ったビルトインの検証ロジックにより、送信前にフォームを検証できます。JavaScript を自作する必要はありません。

  • DAggregateフィルタ
  • Eタイマー設定
  • Fエンティティのツリー

正解：B

Action Flow Editor はアクションのロジックをグラフィカルなフローとして表示するため、条件分岐やループの実行パスを視覚的に追跡・理解しやすくなります。

Q40

エンティティの属性にデフォルト値を設定するにはどうしますか？

  • AJavaScript で設定
  • Bロールで設定
  • CAggregateで設定
  • DCSS で設定
  • E設計時の属性プロパティ（Default Value）で設定
  • F画面で設定

正解：E

Entity 属性のデフォルト値は、設計時に属性のプロパティ（Default Value）で設定します。設定した値は、新規レコード作成時の初期値として使われます。

Q41

あるフィールドを管理者だけが編集できるようにするにはどうしますか？

  • AIf() ロジックと Check<Role>() を使って表示・アクセスを制御する
  • Bすべてのフィールドを表示
  • Cセッション変数を設定
  • Dタイマーを追加
  • Eエンティティを追加
  • Fテーマを追加

正解：A

If() と Check<ロール名>() を組み合わせて、ユーザーのロールに応じてフィールドの表示や編集可否（Enabled）を条件制御します。これにより、管理者ロールを持つユーザーだけに編集を許可できます。

Q42

リストで UI を繰り返して表示するのに最適なオブジェクトはどれですか？

  • Aグリッドスタイル
  • Bタイマー
  • Cイベントベースのロジック
  • Dテーブルレコード（Table）またはリスト（List）ウィジェット
  • Eデフォルトのスタイリング
  • F値のみ

正解：D

リストデータを繰り返し表示するには、Table Records（Table）や List ウィジェットを使います。Aggregate などが返すリストをソースとして、各レコードを行として動的に描画します。

Q43

OutSystems 標準では提供されないカスタム UI を作成する方法はどれですか？

  • Aスタイルシートの変更
  • BAggregateの追加
  • Cエンティティの追加
  • Dタイマーの実行
  • Eロールの追加
  • FWeb ブロックにスクリプトタグを使った JavaScript の利用

標準ウィジェットで実現できないカスタム UI は、JavaScript を使って作成します。スクリプトにより UI の動作や見た目を拡張し、独自のインタラクションを実装できます。

Q44

セッション変数を使用すべきタイミングはどれですか？

  • Aセッション内の別の画面間でデータが必要なとき
  • BAggregate
  • Cローカル変数
  • Dテーマスタイル
  • ECDT（ユーザ定義型）
  • Fエンティティ属性

正解：A

Session Variable は、同一ユーザーのセッション中に複数の画面をまたいでデータを保持したいときに使用します。画面遷移で失われる Local Variable とは異なり、セッションが続く限り値が維持されます。
