Q1

標準の OutSystems アプリ（Users モジュール）で 'Group' を定義するとき、'Group' は主に何に使われますか？

  • AService Studio 内でモジュールをグループ化するため。
  • B複数のユーザーに一度に Roles を割り当てるため。
  • Cチャットグループを作成するため。
  • Dサーバーアクションをグループ化するため。

正解：B

Group は Roles のコンテナ。例えば『IT グループ』に管理者 Role を割り当てると、そのグループの全ユーザーがその Role を継承し、権限管理を効率化できます。

Q2

エンティティ 'Ticket' の 'StatusId' について、Table で ID ではなくラベル（例："Open"）を効率的に表示するには？

  • A行の式で GetStatus(Ticket.StatusId).Label を使う。
  • Bクライアント変数にステータス一覧を保存し JavaScript でルックアップする。
  • Cラベルは表示できず ID のみ表示できる。
  • DTicket 取得の Aggregate に 'Status'（Join Ticket.StatusId = Status.Id）を追加し、Table で 'Status.Label' を使う。

正解：D

Aggregate 内で Join するのが最も効率的（単一クエリ）。行の式でアクションを呼ぶと N+1 クエリ問題（行ごとに別クエリ）となり性能が大きく劣化します。

Q3

ブロックにプレースホルダーがあり、利用者がそこに何もウィジェットを置かなかった場合どうなりますか？

  • A画面を公開できなくなる。
  • Bブロックがエラーをスローする。
  • C何も表示されずスペースが折りたたまれる（プレースホルダーにデフォルトコンテンツが定義されている場合を除く）。
  • Dそのスペースに赤い『X』が表示される。

正解：C

プレースホルダーはオプションのスロット。空なら何も表示せず収縮します。ブロック側でエディタ内にウィジェットを置いておくとデフォルトコンテンツとして表示され、利用者が上書きできます。

Q4

'Service Center' → 'Monitoring' → 'Environment Health' ページの主な目的は？

  • AOutSystems のサービス（Deployment Controller、Scheduler など）とデータベースの接続状況を確認する。
  • Bアプリのソースコードを閲覧する。
  • C新しいユーザーを作成する。
  • Dクライアントの PC の CPU 使用率を監視する。

正解：A

主要システムコンポーネントの状態を信号色（赤/黄/緑）で示します。『タイマーが動かない』時にまず確認する場所で、Scheduler が赤なら原因が分かります。

Q5

Container ウィジェットをブロックで使う有効なシナリオはどれですか？

  • A複数のウィジェットをグループ化して CSS クラスを適用したり 'If' で表示制御したりする。
  • B親画面からウィジェットを渡せるようにする。
  • C画面にコメントを追加する。
  • Dブロックに入力変数を作成する。

正解：A

Container は div としてレンダリングされる構造ウィジェットで、コンテンツをまとめてスタイル/ロジックを適用します。親がコンテンツを注入する『穴』は Placeholder です。

Q6

'Employee' と 'City' を結合し、City が割り当てられていない Employee も含めて全て取得したい。どの結合タイプ？

  • AWith or Without（左外部結合）
  • BOnly With（内部結合）
  • CCross Join（クロス結合）
  • DWith（完全外部結合）

正解：A

'With or Without' は SQL の Left Join に相当し、左（Employee）の全レコードを返し、City があれば結合、なければ null。'Only With'（Inner Join）は両方一致のみで City 無しは除外されます。

Q7

アーキテクチャキャンバスで『循環依存関係』の違反を構成するのは何ですか？

  • Aモジュール A と B が両方ともモジュール C から要素を消費している。
  • Bモジュール A が自分自身を参照している。
  • Cモジュール A が B の公開要素を消費し、B が A の公開要素を消費している（直接または間接的に）。
  • Dモジュール A が B を消費し、B が C を消費している。

正解：C

循環依存は2つのモジュールが互いを参照し合う場合に発生し、密結合・独立展開不可・壊れた参照を招きます。A→B、B→C は正常な階層です。

Q8

エンティティの 'Text' 属性の最大長は、特に指定しない場合デフォルトで何文字ですか？

  • A100文字
  • B無制限（最大）
  • C50文字
  • D255文字

正解：C

新しい Text 属性を作ると既定の Length は 50 です。必要に応じ手動で増やします（増やし忘れると 50 文字で切り捨てられるのが典型的なバグ要因）。

Q9

Reactive Web App のシステムイベント 'On Application Ready' の目的は？

  • A画面がロードされるたびに実行される。
  • Bサーバーが起動したときに実行される。
  • Cユーザーがログインしたときに実行される。
  • Dブラウザにアプリがロードされたときに一度だけ実行され、グローバルな設定処理（例：アナリティクス設定、初期トークン確認）が可能。

正解：D

OnApplicationReady はクライアント側アプリの『Main()』的エントリポイントで、最初の画面がレンダリングされる前に一度だけ実行され、サードパーティスクリプト設定やグローバル既定値の設定に使います。

Q10

『Static Entities（静的エンティティ）』について誤っている記述はどれですか？

  • AStatic Entities はデータベースに格納されている。
  • B'Create' アクションを使って実行時に新しいレコードを追加できる。
  • CStatic Entities はリテラル識別子（例：Status.New）を持つ。
  • DStatic Entities は外部キーとして使用できる。

正解：B

誤りはこれ。静的エンティティは実行時に変更できません。レコードは Service Studio で管理され、公開（デプロイ）時に更新されます（列挙型のような定数扱い）。

Q11

'JSON Deserialize' ツールの 'Data Type' プロパティに必要な必須要件は？

  • A'Text' または 'Integer' タイプのみを受け付ける。
  • BJSON 文字列から構造を自動的に検出する。
  • CXML スキーマが必要。
  • DJSON 構造に一致する Structure、Entity、または List 定義を提供する必要がある。

正解：D

OutSystems は静的型付けのため、デシリアライズ先の型を設計時に定義する必要があります。受信 JSON のプロパティを変数にマッピングする設計図（Structure 等）が必須です。

Q12

ウィジェットの 'Style Classes'（スタイルクラス）プロパティからテーマの CSS クラスを参照する正しい構文は？

  • A".my-class"（ドット付き）
  • Bクラス名をそのまま入力（例：my-class）
  • CClass("my-class")
  • D"#my-class"

正解：B

Style Classes は HTML の class 属性に直接対応し、クラス名だけ（ドット/#/引用符なし）を、複数ならスペース区切りで書きます。ドットは CSS 定義側、# は ID 用です。

Q13

サーバーとは異なるタイムゾーンで日時を表示したいとき、最も役立つのは？

  • A単純に5時間引き算する。
  • Bサーバーのタイムゾーン設定を変更する。
  • C'Time Zone' Forge コンポーネント（または利用可能なら ConvertFromTimeZone 関数）。
  • D'FormatDate' 関数を使う。

正解：C

タイムゾーン変換は DST を含め複雑なので、Forge のタイムゾーンコンポーネント等で IANA タイムゾーンに基づき変換します。オフセットのハードコードは DST に対応できず壊れやすいです。

Q14

ブロックに Boolean の入力パラメータ 'IsActive' があります。これに応じてブロック内コンテナのスタイルを変える最適な方法は？

  • A状態ごとに 'If' で囲んだ2つの異なるコンテナを使う。
  • Bコンテナの 'Style Classes' に式 If(IsActive, "status-active", "status-inactive") を使う。

正解：B

Style Classes は式をサポートします。条件で CSS クラスを切り替えると HTML はクリーンでスタイルは CSS に集約できます。DOM 重複やインラインスタイルは避けます。

Q16

クライアントアクション内でサーバーアクションを呼び出した場合、デフォルトの実行の流れは？

  • Aサーバーアクションは後で実行されるようキューに追加される。
  • Bブラウザに依存する。
  • Cクライアントアクションはサーバーアクションの完了まで待機し（開発者視点で同期的に振る舞い）、次のノードに進む。
  • Dクライアントアクションはすぐ続行し、サーバーアクションはバックグラウンドで実行される。

正解：C

Q17

モジュール公開時に 'Upgrade Violation' エラーが 'Strong Dependency' メッセージとともに表示されました。何を意味しますか？

  • Aプロデューサーモジュールが削除された。
  • Bサーバーアクションを多く使いすぎている。
  • C循環依存がある。
  • Dプロデューサーの公開アクションの入出力シグニチャを変更したが、コンシューマーをまだリフレッシュ/再公開していない。

正解：D

強い依存の署名が変わると、コンシューマーは存在しない/異なる関数を呼ぼうとします。コンシューマーを開いて『依存関係をリフレッシュ』し再公開する必要があります（削除は Missing Reference）。

Q19

Excel からのブートストラップ機能で、Excel に重複行が含まれていた場合どうなりますか？

  • A自動的に重複をマージする。
  • Bすぐにインポートを停止する。
  • Cエンティティに定義されたインデックス次第。ユニークインデックスがあれば重複は失敗し、なければ挿入される。
  • D既存レコードを更新する。

正解：C

ブートストラップは基本的に Create 動作で、DB の制約に従います。エンティティにユニークインデックスがなければ重複レコードも挿入され得ます（自動で重複排除する仕組みはありません）。

Q20

リアクティブ Web アプリの検索で、ユーザーが入力を止めてから500ms後に検索したい（Debounce）。どう実現しますか？

  • AInput の OnChange で setTimeout を使う JavaScript ノードを用い、既存のタイムアウトをクリアする。
  • BInput ウィジェットの 'Debounce' プロパティを使う。
  • C500ms ごとに動く Timer を使う。
  • DOnChange 内で 'Sleep' アクションを使う。

正解：A

Debounce は setTimeout/clearTimeout を使うクライアント側 JS で実装するのが標準。標準 Input に Debounce プロパティはなく、Timer はサーバー側背景処理、Sleep は UI をフリーズさせます。

Q21

ユーザーがログアウトしたとき、'セッション変数' のデータはどうなりますか？

  • Aセッションが無効になり、次のセッションのためにセッション変数はデフォルト値にリセットされる。
  • Bデータはブラウザを閉じるまで保持される。
  • Cデータは自動的にデータベースに保存される。
  • Dデータは次のユーザーのためにコンピュータ上に残る。

正解：A

セッション変数はユーザーのアクティブなセッションに紐づく揮発性の状態。ログアウト（User_Logout）やタイムアウトでセッションが破棄され、次のセッションでは既定値に戻ります。

Q22

Switch ウィジェットに Status="New"、Status="Closed"、Otherwise の3ブランチがあります。Status が "New" のとき、どのブランチが実行されますか？

  • AOtherwise がデフォルトなので Otherwise ブランチが実行される。
  • BNew ブランチが実行され、その後 Otherwise にフォールスルーする。
  • C条件が最初に True になるブランチだけが実行される（この場合 Status="New"）。
  • DTrue と評価される全ブランチが実行される。

正解：C

Switch は上から順に条件を評価し、最初に True になったブランチだけを実行して評価を止めます（If-ElseIf-Else のような排他選択。フォールスルーはしません）。

Q23

GrantRole と CheckRole の主な違いは何ですか？

  • AGrantRole は権限をデータベースに保存し、CheckRole は現在のリクエストで権限を確認する。
  • BGrantRole は一時的で、セッション期間だけ有効。
  • CGrantRole は画面向け、CheckRole はロジック向け。
  • D違いはない。

正解：A

Grant は書き込み（権限の永続的な割り当て）、Check は読み取り（検証ロジック）です。Grant は管理操作、Check はセキュリティガードとして都度権限を検証します。

Q24

特定のロール（例：'Approver'）をプログラム的にユーザーに付与するには、どのアクションを使いますか？

  • Aユーザーエンティティを直接更新する。
  • B'CreateUserRole' エンティティアクション。
  • C'AddRoleToUser' システムアクション。
  • D'GrantApproverRole' サーバーアクション。

正解：D

'Approver' ロールを定義すると、プラットフォームは自動的に 'GrantApproverRole' / 'RevokeApproverRole' / 'CheckApproverRole' を生成します。Grant<RoleName>Role(UserId) が型安全で標準的な付与方法です。ロールは別のリンクテーブルに保存され、ユーザーレコードに直接は保存されません。
